## Multi-Platform and Team Configuration

Suncode ships on 18 platforms (Claude Code, Cursor, OpenCode, Engineer, Codex, Kiro, Kilo, Gemini CLI, Antigravity, Devin (formerly Windsurf), Qoder, CodeBuddy, GitHub Copilot, Droid, Pi Agent, Reasonix, ZCode, Trae) and additionally supports any AI coding agent that reads the `.agents/skills/` standard (Amp, Cline, Deep Agents, Firebender, Kimi Code CLI, Warp, and more). The `.suncode/` core is identical everywhere; what differs is **how hooks, extensions, skills, sub-agents, and commands are delivered** on each platform.

### Joining an Already-Initialized Suncode Project

Someone else already ran `suncode init` on the repo and you're joining as a new team member. Just run `suncode init` â€” the CLI detects the existing setup and offers three choices:

```
? Suncode is already initialized. What would you like to do?
âťŻ Add AI platform(s)
  Set up developer identity on this device
  Full re-initialize
```

| Option                                       | When to pick                                                                                                    |
| -------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| **Add AI platform(s)**                       | You want to add a platform others haven't configured (e.g. you use Cursor but the team only set up Claude Code) |
| **Set up developer identity on this device** | You're a new member, you just need to write your **identity** on your machine                                   |
| **Full re-initialize**                       | Project config is broken and you want a clean slate                                                             |

New members pick the second option. The CLI asks for your developer name (defaulting to your Git user), then:

- Writes `.suncode/.developer` with your developer name (gitignored, per-machine)
- Creates `.suncode/workspace/<your-name>/` for your own journal

After that, on platforms with a SessionStart hook or extension, opening a new session auto-injects Suncode context. On platforms without automatic session injection, run `/suncode:start` or the platform's start workflow.

<Note>
  Don't pick **Full re-initialize** â€” it overwrites existing `.suncode/`, `.claude/`, etc.
  configuration and affects the whole team.
</Note>

### Claude Code

Most complete automation. Hook layout:

| Hook                         | Trigger           | Effect                                         |
| ---------------------------- | ----------------- | ---------------------------------------------- |
| `session-start.py`           | SessionStart      | Injects identity, git status, active tasks     |
| `inject-workflow-state.py`   | UserPromptSubmit  | Nudges the AI toward the current task's state  |
| `inject-subagent-context.py` | PreToolUse (Task) | Loads `implement.jsonl` / `check.jsonl` / etc. |

Sub-agents: `suncode-implement`, `suncode-check`, `suncode-research` under `.claude/agents/`.
Skills: `suncode-brainstorm`, `suncode-before-dev`, `suncode-check`, `suncode-update-spec`, `suncode-break-loop` under `.claude/skills/`.
Commands: `start`, `finish-work`, `continue` under `.claude/commands/suncode/`.

### Cursor

```bash
suncode init -u your-name --cursor
```

Cursor is a full class-1 platform: real hooks, real sub-agents, real skills.

Layout:

- `.cursor/commands/suncode-{name}.md`: `start`, `finish-work`, `continue` (flat file naming with `suncode-` prefix, invoked as `/suncode-start` etc.)
- `.cursor/skills/suncode-{name}/SKILL.md`: the 5 suncode skills
- `.cursor/agents/`: `suncode-implement.md`, `suncode-check.md`, `suncode-research.md`
- `.cursor/hooks/`: shared Python hook scripts (Claude-Code-compatible)
- `.cursor/hooks.json`: hook configuration (Cursor uses a separate file, not `settings.json`)

### OpenCode

```bash
suncode init -u your-name --opencode
```

OpenCode 1.2.x is a class-1 platform (real hooks + real sub-agents):

- `.opencode/commands/suncode/`: start / finish-work / continue
- `.opencode/agents/`: `suncode-implement.md`, `suncode-check.md`, `suncode-research.md`
- `.opencode/skills/`: the 5 suncode skills
- `.opencode/plugins/`: JS plugins: `session-start.js`, `inject-subagent-context.js`, `inject-workflow-state.js`

Plugins are factory functions (OpenCode 1.2+). Suncode generates them so behavior parity with Claude Code is the goal.

### Engineer

### Codex

```bash
suncode init -u your-name --codex
```

Layout:

- `AGENTS.md` (repo root): entry file; Codex auto-reads it every session (acts as the prelude)
- `.codex/prompts/`: `suncode-start.md`, `suncode-finish-work.md`, `suncode-continue.md`
- `.codex/skills/`: the 5 suncode skills
- `.codex/agents/`: TOML sub-agents: `suncode-implement.toml`, `suncode-check.toml`, `suncode-research.toml`
- `.codex/hooks/inject-workflow-state.py` + `.codex/hooks.json`: `UserPromptSubmit` workflow-state hook
- `.codex/hooks/session-start.py`: retained compact SessionStart compatibility script, not wired by default

<Note>
Codex hooks require enabling. Add to `~/.codex/config.toml`:

```toml
[features]
hooks = true   # Codex 0.129+. Older versions: `codex_hooks = true`.
```

Codex 0.129+ also gates each installed hook behind a one-time `/hooks` TUI review â€” run `/hooks` once in Codex and approve the Suncode `UserPromptSubmit` hook, otherwise the workflow breadcrumb won't auto-inject. Without these, Codex falls back to prelude-only (reads `AGENTS.md` every session): same content, just no event-driven injection.

</Note>

### Kiro

```bash
suncode init -u your-name --kiro
```

Layout:

- `.kiro/agents/suncode.json`: main Suncode agent â€” per-turn `userPromptSubmit` hook + session-start `agentSpawn` hook + `.suncode/workflow.md` as an always-loaded resource
- `.kiro/agents/suncode-{implement,check,research}.json`: sub-agents (`agentSpawn` injects task context)
- `.kiro/hooks/*.py` + `.kiro/hooks/suncode-workflow-state.kiro.hook`: the per-turn workflow-state injector (CLI agent hook + IDE `.kiro.hook`)
- `.kiro/skills/*/SKILL.md`: auto-trigger skills

**Enable it (required â€” otherwise the workflow won't activate):**

- **Kiro CLI**: make `suncode` the active agent so its hooks fire â€” `kiro-cli settings chat.defaultAgent suncode` (persists) or `/agent swap suncode` (per session). Kiro otherwise runs the built-in `kiro_default` agent.
- **Kiro IDE**: the `.kiro/hooks/suncode-workflow-state.kiro.hook` (a `promptSubmit` hook) ships enabled; confirm it's on/trusted in Kiro's Agent Hooks UI.

<Note>
  The per-turn injection prints plain text that Kiro adds to the conversation context, per Kiro's
  official hooks docs. The exact stdout-to-context behavior (and whether the IDE `runCommand` action
  injects stdout) is pending verification on real Kiro hardware; if it doesn't, the fallback is a
  static steering nudge.
</Note>

### Qoder

### Pi Agent

```bash
suncode init -u your-name --pi
```

Pi Agent is extension-backed rather than Python-hook-backed. Suncode writes the same workflow primitives, then the extension resolves the current session id and injects task context before Bash commands and sub-agent runs.

Layout:

- `.pi/prompts/suncode-{name}.md`: session-boundary prompts (`finish-work`, `continue`; `start` only where applicable)
- `.pi/skills/suncode-{name}/SKILL.md`: the five Suncode workflow skills
- `.pi/agents/{name}.md`: `suncode-implement`, `suncode-check`, `suncode-research`
- `.pi/extensions/suncode/index.ts`: session context, Bash `SUNCODE_CONTEXT_ID` propagation, and sub-agent JSONL injection
- `.pi/settings.json`: extension registration

The extension stores active task state under `.suncode/.runtime/sessions/<session-key>.json`, so each Pi window/session can work on its own task without taking over another window.

### Other supported platforms

Suncode also ships configurators for platforms not covered by the capability matrix:

- **Kilo** (`--kilo`): writes `.kilocode/workflows/` (commands: `start`, `finish-work`) and `.kilocode/skills/suncode-{name}/SKILL.md` (the 5 suncode skills). No hook integration.
- **Antigravity** (`--antigravity`): writes Antigravity-native workflow files for the three commands.
- **Devin** (`--devin`, formerly Windsurf): writes Devin-native workflow and skill files. The old `--windsurf` flag still works as a deprecated alias.
- **Reasonix** (`--reasonix`): writes `.reasonix/skills/` for workflow skills plus `runAs: subagent` implement/check skills. No hook integration.
- **ZCode** (`--zcode`): writes `.zcode/commands/suncode/`, `.zcode/cli/agents/`, and `.agents/skills/`. Sub-agents use a pull-based prelude.
- **Trae** (`--trae`): writes `.trae/commands/`, `.trae/skills/`, `.trae/agents/`, `.trae/hooks/`, and `.trae/hooks.json`. Main-session context uses SessionStart/UserPromptSubmit hooks; sub-agents use a pull-based prelude.

Beyond the 18 configured platforms, Suncode can be consumed by any AI coding agent that follows the `.agents/skills/` convention (the [agentskills.io](https://agentskills.io) standard). Codex writes its skills there, and the files are directly usable by other agents in that ecosystem (Amp, Cline, Deep Agents, Firebender, Kimi Code CLI, Warp, and more). On those platforms you manage Suncode through the `.suncode/` core plus whatever prelude file the agent reads.

### Multi-Developer Collaboration

Per-developer isolation (no conflicts):

- `.suncode/workspace/{name}/`: each developer's own journals and index
- `.suncode/.developer`: gitignored
- `.suncode/.runtime/`: gitignored session runtime; each AI session/window has its own active task file

Shared state (coordinate via PR):

- `.suncode/spec/`: team conventions, PR-reviewed like any code
- `.suncode/tasks/`: task JSONs; explicit `--assignee` avoids collisions

Important spec changes should be discussed in reviews; treat the spec library as team code.

### `suncode update` and Version Management

```bash
cat .suncode/.version          # current version
suncode update                  # update to latest
suncode update --dry-run        # preview
suncode update --migrate        # apply breaking-change migrations (required for major)
suncode update -f               # force overwrite locally-modified files
suncode update -s               # skip locally-modified files
```

Template hash mechanism (`.suncode/.template-hashes.json`):

1. Compute local file hash.
2. Compare against recorded template hash.
3. Match â‡’ file unchanged by user â‡’ safe to update.
4. Differ â‡’ prompt (overwrite / skip), or silently respect policy with `-f` / `-s`.

Breaking changes (for example, removing the Multi-Agent Pipeline in 0.5.0) ship as migration manifests: running `suncode update` without `--migrate` on a breaking-manifest exits with instructions instead of silently renaming files. `suncode update --migrate` applies the rename / delete entries, asking once per conflict.
