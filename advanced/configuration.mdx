## Configure `.suncode/config.yaml`

`.suncode/config.yaml` is the project-level configuration file for Suncode runtime behavior. It controls session journal commits, task lifecycle hooks, optional Hub team collaboration, package mapping, and Codex dispatch mode.

Track this file with the repo. Treat it as shared team configuration, not as a place for per-machine identity or secrets. Per-machine identity belongs in `.suncode/.developer`. Hub sign-in is managed by `suncode hub login`; do not put Hub credentials in `.suncode/config.yaml`.

## Edit safely

Most sections are optional. Leave commented examples commented until you need them, and edit only the keys your project actually uses.

Use normal YAML values:

```yaml
session_commit_message: 'chore: record journal'
max_journal_lines: 2000
session_auto_commit: false
```

When a key is absent, Suncode uses its built-in default. Keep indentation consistent for nested blocks such as `hooks`, `packages`, and `codex`.

<Warning>
  Hook commands live in tracked project configuration. Do not put credentials, secrets, or
  machine-specific absolute paths in `.suncode/config.yaml`.
</Warning>

## Current keys

| Key                      | Default                   | What it controls                                                                                                  |
| ------------------------ | ------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| `session_commit_message` | `"chore: record journal"` | Commit message used by `add_session.py` when it auto-commits journal and index changes.                           |
| `max_journal_lines`      | `2000`                    | Maximum lines per journal file before Suncode rotates to the next journal file.                                   |
| `session_auto_commit`    | `true`                    | Whether `add_session.py` and `task.py archive` auto-stage and auto-commit Suncode journal / task archive changes. |
| `hooks`                  | unset                     | Shell commands that run after task lifecycle events.                                                              |
| `hub`                    | disabled                  | Optional team collaboration block for binding local Suncode tasks to Suncode Hub work.                            |
| `packages`               | unset                     | Package declarations for monorepos, submodules, and polyrepo-style directories.                                   |
| `default_package`        | unset                     | Fallback package when a command or task does not specify one.                                                     |
| `update.skip`            | unset                     | Legacy update exclusion list that `suncode update` still honors during normal updates.                            |
| `codex.dispatch_mode`    | `inline`                  | Codex-only choice between main-agent implementation and legacy `suncode-*` sub-agent dispatch.                    |

### Session recording

`session_commit_message` and `max_journal_lines` apply to session journal recording:

```yaml
session_commit_message: 'chore: record journal'
max_journal_lines: 2000
```

Use `session_commit_message` if your repo has a conventional commit style for Suncode journal commits. Use `max_journal_lines` to keep `.suncode/workspace/<developer>/journal-N.md` files at a reviewable size.

### Session auto-commit

`session_auto_commit` controls whether Suncode scripts touch git for journal and task archive bookkeeping:

```yaml
session_auto_commit: false
```

Default `true` preserves the normal behavior: `add_session.py` and `task.py archive` write files, stage the Suncode changes, and create bookkeeping commits. Set it to `false` when `.suncode/` is intentionally gitignored or when your team wants to review and commit Suncode bookkeeping manually.

Accepted values are `true`, `false`, `yes`, `no`, `1`, `0`, `on`, and `off` (case-insensitive). Invalid values fall back to `true` and print a warning.

### Task lifecycle hooks

`hooks` runs shell commands after task lifecycle events. Each command receives `TASK_JSON_PATH`, pointing at the task's `task.json`.

```yaml
hooks:
  after_create:
    - "echo 'Task created'"
  after_start:
    - "echo 'Task started'"
  after_finish:
    - "echo 'Task finished'"
  after_archive:
    - "echo 'Task archived'"
```

Supported events are `after_create`, `after_start`, `after_finish`, and `after_archive`. Hook failures print a warning but do not block the main task operation.

### Hub team collaboration

`hub` enables optional team collaboration for projects connected to Suncode Hub. It is disabled by default. Local-only Suncode projects do not contact Hub and do not need Hub sign-in.

Use `suncode hub init` when possible; it writes the project block and the matching user-level default configuration for you.

```yaml
hub:
  enabled: true
  mode: team
  projectId: your-project-id
  developerId: null
  apiBaseUrl: null
  startReviewPolicy: confirm
  review:
    enabled: false
    provider: engineer
    required: false
    trigger: manual
    unavailablePolicy: bypass
    engineer:
      command: engineer
      args: ['run']
      timeoutSeconds: 900
      saveRawOutput: true
  sync:
    afterCreate: true
    afterStart: true
    afterFinish: false
    afterArchive: true
```

| Key                                 | Meaning                                                                                                            |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| `hub.enabled`                       | `true` opts this project into Hub team mode. Any other value keeps the project local.                              |
| `hub.mode`                          | Current supported value is `team`.                                                                                 |
| `hub.projectId`                     | Team-provided project identifier used to bind local tasks to the right Hub project.                                |
| `hub.developerId`                   | Optional override. Leave `null` to use the signed-in Hub identity or local developer ID.                           |
| `hub.apiBaseUrl`                    | Optional project override. Leave `null` to use the user-level Hub default.                                         |
| `hub.startReviewPolicy`             | `confirm`, `block`, or `bypass` for start-review gating on Hub tasks.                                              |
| `hub.review.enabled`                | Enables explicit `suncode hub review` rounds for Hub-bound tasks.                                                  |
| `hub.review.provider`               | Review provider. The first supported provider is `engineer`.                                                       |
| `hub.review.required`               | Default `false`. Set `true` to block `submit-completion` until the latest review is approved for the current diff. |
| `hub.review.trigger`                | Default `manual`; run review explicitly with `suncode hub review`.                                                 |
| `hub.review.unavailablePolicy`      | `bypass`, `warn`, or `block` when the configured review provider is unavailable.                                   |
| `hub.review.engineer.saveRawOutput` | Default `true`; set `false` to avoid storing the provider's original stdout/stderr in `raw-output.md`.             |
| `hub.sync.afterCreate`              | Sync task creation to Hub when a Hub-bound task is created.                                                        |
| `hub.sync.afterStart`               | Sync structured subtasks and start state when a Hub-bound task starts.                                             |
| `hub.sync.afterFinish`              | Reserved for teams that want a finish hook before archive. Default is `false`.                                     |
| `hub.sync.afterArchive`             | Submit completion state when a Hub-bound task is archived.                                                         |

Hub review is task-centered and CLI-orchestrated. When review is enabled, run:

```bash
suncode hub review --task current
```

Suncode creates the next `reviews/round-NNN/` directory, captures the current diff, invokes the provider, writes `review.json` and a concise `result.md` review report, uploads review artifacts through the existing Hub artifact upload flow, and syncs task status. When `saveRawOutput` is true, Suncode also keeps the provider transcript in `raw-output.md` for diagnostics. The AI should read the result and fix `changes_requested` items; it should not hand-maintain review status or uploads.

Hub sign-in is separate from project configuration:

```bash
suncode hub login
suncode hub state
```

Never store Hub credentials or private machine-specific values in `.suncode/config.yaml`. If `hub.enabled` is `true`, run `suncode hub state` to confirm the project is configured and signed in before starting Hub-specific work.

### Packages and default package

`packages` declares project structure for monorepos and repos with multiple working directories:

```yaml
packages:
  frontend:
    path: packages/frontend
  backend:
    path: packages/backend
  docs:
    path: docs-site
    type: submodule
  webapp:
    path: ./webapp
    git: true

default_package: frontend
```

Use `type: submodule` for git submodules. Use `git: true` when a subdirectory is its own independent git repository, as in polyrepo or meta-repo layouts.

`default_package` must match a key under `packages`. Suncode uses it when a task or command does not specify a package.

### Update skip compatibility

`update.skip` is still supported for projects that already use it, even though the current template no longer includes a commented example:

```yaml
update:
  skip:
    - .claude/commands/
    - docs-site/
```

During normal `suncode update` runs, skipped paths are excluded from template writes and safe-file-delete cleanup. During breaking migrations that you run with `--migrate`, Suncode can bypass `update.skip` for migration-required cleanup so the project does not stay half-updated; the update command prints a warning before doing this.

### Codex dispatch mode

`codex.dispatch_mode` is only read by Codex workflows. Other platforms ignore it.

```yaml
codex:
  dispatch_mode: inline # or "sub-agent" to dispatch suncode-* sub-agents
```

`inline` keeps implementation and checking in the main Codex agent. `sub-agent` opts into the older dispatch model where the main agent launches `suncode-implement`, `suncode-check`, or `suncode-research` sub-agents.

The default is `inline` because Codex sub-agents run in isolated turns and cannot inherit the parent session's full task context. Use `sub-agent` only when you explicitly want that legacy split.

## Update behavior

`suncode init` writes the current template for new projects. For existing projects, `suncode update` preserves local edits to `.suncode/config.yaml` instead of replacing the file wholesale.

When a release adds a new config section, the migration manifest can declare it through `configSectionsAdded`. During `suncode update`, Suncode checks whether the configured sentinel text already appears in the target file. If the sentinel is missing, Suncode appends the matching section from the bundled template to the end of `.suncode/config.yaml`.

This append-only path is idempotent and preserves your existing values. The `session_auto_commit` section was delivered this way for existing projects: if `session_auto_commit:` was missing, `suncode update` appended the commented section; if it was already present, update skipped it.

After an update, review any appended commented section and uncomment only the keys you want to activate.
