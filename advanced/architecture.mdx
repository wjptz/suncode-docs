Trellis is a **team-level agent harness with a built-in LLM wiki** — an engineering shell wrapping AI coding plus an in-repo knowledge base the LLM reads directly. This page is for developers who want to understand how the system runs and where to extend. End-user usage flow lives at [How it works](/start/how-it-works).

## Design Philosophy

Trellis treats AI coding as an engineering problem, not a chat problem.

- **No reliance on memory** — specs are pushed into hooks / sub-agent preludes; the AI doesn't need to "remember" to read them
- **No reliance on conversation** — knowledge lives in files (`spec/` / `tasks/` / `workspace/`); compacted conversations don't lose state
- **No reliance on a single super-agent** — work is split across sub-agents with bounded scope and clear responsibility
- **No reliance on trust** — the AI does not commit; the commit boundary stays with humans
- **No reliance on generic answers** — team conventions are project-specific corpus, not Stack Overflow

## Feature Overview

Three concurrent threads derive from the vision: harness governs AI behavior, wiki governs team knowledge, and the team-level dimension amplifies both.

### Agent Harness (engineering shell that constrains AI behavior)

- **Native support across 14+ AI coding platforms** — Claude Code / Cursor / Codex / Gemini / Qoder / OpenCode / Copilot / CodeBuddy / Droid / Pi / Kiro / Kilo / Antigravity / Windsurf. Each platform uses its native hook / skill / extension / workflow primitive; the same `.trellis/` produces equivalent behavior everywhere
- **Three-Phase workflow state machine** — Plan / Execute / Finish, with a per-turn breadcrumb injected into the main session at every step. When the AI tries to skip a step, the hook pulls it back
- **Three-tier sub-agent execution** — `trellis-research` (investigate) / `trellis-implement` (write code) / `trellis-check` (verify and self-fix); separation of concerns plus mutual constraint
- **Mandatory read-before-write** — three mechanisms (`before-dev` skill / `implement.jsonl` injection / pull-based prelude) ensure sub-agents read the relevant spec before writing code
- **AI does not commit** — harness boundary: sub-agents finish, humans verify and commit; every change goes through PR review

### Built-in LLM Wiki (team knowledge that the AI reads automatically)

- **`.trellis/spec/`** — team conventions library; organized by package and layer; sub-agents pull on demand via the task's `implement.jsonl`, never dumped wholesale into the main session
- **`.trellis/tasks/`** — task-level knowledge; each task carries its own `prd.md` + `research/*.md` + `info.md` + jsonl context manifests
- **`.trellis/workspace/`** — developer-level journals; cross-session retro, decision log, prevent re-stepping on the same rake
- **`.trellis/workflow.md`** — the workflow's own wiki; single source of truth, edit one file and the whole project's workflow shifts

### Team-level (collaboration dimension)

- **`.trellis/` belongs to the team, not the individual** — workflow / spec / tasks are all git-tracked; a new developer `git clone`s and gets the full harness plus team knowledge base; the first `trellis init` auto-creates a joiner task to onboard
- **Multi-task parallelism + subtask trees** — multiple tasks running in the same repo at once; subtask branches roll up to the parent; the active-task pointer is session-scoped so multiple windows don't cross-talk
- **Cross-developer knowledge accrual** — PRDs and research someone wrote into a task, products archived under `tasks/archive/`, conventions distilled into `spec/` — every team member's sub-agents can read them next time (not just the original author)
- **Per-developer workspace** — `.trellis/workspace/<dev>/` isolates each developer's journals so they don't overwrite each other; `spec/` remains the team's shared SoT

## `workflow.md`: Single Source of Truth

`.trellis/workflow.md` is the only definition of the Plan / Execute / Finish contract. All steps across the three phases (1.0 `task.py create` → 1.4 `task.py start` → 2.1 `trellis-implement` → 2.2 `trellis-check` → 3.4 `/trellis:finish-work`) live here. Every other system component reads from it:

- **SessionStart hook** auto-injects its TOC + step bodies, so the AI has the contract before you say anything
- **`/trellis:continue`** reads the current session runtime (`.trellis/.runtime/sessions/<session-key>.json`) plus the task's `status`, then matches against `workflow.md` to decide which step you're on and what's next
- **Per-turn `<workflow-state>` breadcrumbs** (`inject-workflow-state.py` hook) fire on every user message and inject the `[workflow-state:STATUS]` block matching the active task's `status`. The hook is parser-only; body text comes verbatim from `workflow.md`. When a status has no matching block, the hook emits `Refer to workflow.md for current step.` instead.
- **Sub-agent injection** — `workflow.md` is the default first entry of `implement.jsonl`, so sub-agents see the contract too

**Default task statuses**: `planning` / `in_progress` / `completed`, written by `task.py create` / `start` / `archive` respectively, aligned 1:1 with the three phases. `task.py create` also sets the active-task pointer for the current session, so `[workflow-state:planning]` fires immediately. `workflow.md` carries one `[workflow-state:STATUS]` block per status. Adding a custom status (e.g. `review` / `blocked`) means adding a `[workflow-state:STATUS]...[/workflow-state:STATUS]` block — no code changes.

> Edit `workflow.md` and the entire project's workflow changes. Next session, every platform / skill / sub-agent / command picks up the new version. There's no workflow logic hardcoded elsewhere.

## Spec Injection Mechanism

<Info>
  Sub-agent automatic spec injection depends on the platform's PreToolUse hook or extension
  equivalent ("hook-push"). Codex / Copilot / Gemini / Qoder go through pull-based prelude — the
  sub-agent reads the context itself on its first turn. Both paths sit on shared infrastructure.
</Info>

### Hook-push flow (Claude Code / Cursor / OpenCode / Kiro / CodeBuddy / Droid)

```
Main session spawns: Task(subagent_type="trellis-implement", prompt="...")
     │
     ▼
┌───────────────────────────────────────────────────────────────┐
│  Hook / extension: inject-subagent-context                    │
│  (runs BEFORE the sub-agent starts)                           │
│                                                               │
│  Reads happen in the hook, not the sub-agent:                 │
│                                                               │
│  1. Read current session runtime                              │
│     → .trellis/.runtime/sessions/<session-key>.json           │
│     → current task directory path                             │
│                                                               │
│  2. Read {task_dir}/implement.jsonl                           │
│     → list of spec files this sub-agent needs                 │
│                                                               │
│  3. Read each file referenced by the JSONL                    │
│     → .trellis/spec/backend/index.md                          │
│     → .trellis/spec/backend/database-guidelines.md            │
│                                                               │
│  4. Read prd.md (requirements) and info.md (design, optional) │
│                                                               │
│  5. Concatenate and pass as additionalContext to the sub-agent│
└───────────────────────────────┬───────────────────────────────┘
                                │
                                ▼
       Sub-agent starts with everything in its prompt — it reads
       no files itself.
```

### Pull-based prelude flow (Codex / Copilot / Gemini / Qoder)

These platforms can't reliably rewrite a sub-agent's prompt via hooks (upstream limitation). Instead, the sub-agent definition starts with a prelude: on its first turn, the sub-agent reads `.trellis/.runtime/sessions/<session-key>.json` → `prd.md` → `implement.jsonl`, then reads each file referenced. Result is equivalent to hook-push, just one turn later.

### JSONL Format

JSONL (JSON Lines) tells the hook which files to inject for a sub-agent. One JSON object per line:

```jsonl
{"file": ".trellis/workflow.md", "reason": "Project workflow and conventions"}
{"file": ".trellis/spec/backend/index.md", "reason": "Backend coding rules"}
{"file": "src/services/auth.ts", "reason": "Existing auth pattern to align with"}
{"file": ".trellis/tasks/02-27-user-login/research/", "type": "directory", "reason": "Research outputs"}
```

| Field    | Required | Description                                                                                                                                                                          |
| -------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `file`   | yes      | Path to file or directory (relative to project root)                                                                                                                                 |
| `reason` | yes      | Why the file is needed                                                                                                                                                               |
| `type`   | no       | Default `"file"`. Single-file mode reads any extension (source code, JSON, etc.). `"directory"` mode injects `.md` files in the directory sorted by name (max 20); non-`.md` skipped |

**Three JSONL files**:

| File              | Consumer                      | Typical contents                                       |
| ----------------- | ----------------------------- | ------------------------------------------------------ |
| `implement.jsonl` | `trellis-implement`           | workflow.md + relevant specs + code pattern references |
| `check.jsonl`     | `trellis-check`               | quality checklist specs + test / lint rules            |
| `research.jsonl`  | `trellis-research` (optional) | project structure overview + research goals            |

What goes in JSONL depends on what the sub-agent needs to do this task:

- **Spec and workflow**: `.trellis/workflow.md`, `.trellis/spec/**/index.md`, specific guideline files — shape **how** the sub-agent writes
- **Existing code as reference**: point at single source files (`src/services/auth.ts`) to align with existing patterns. Prefer single files
- **Research outputs**: directory mode pointing at `{task_dir}/research/` pulls every `.md` produced by `trellis-research`

## Sub-agent System

Trellis 0.5 ships three sub-agents. Each has a clear role, clear constraints, and an independent JSONL spec injection channel.

| Sub-agent           | Role                                        | Constraint                              | Primary context              |
| ------------------- | ------------------------------------------- | --------------------------------------- | ---------------------------- |
| `trellis-research`  | Code / docs search, technical investigation | Read-only                               | `research.jsonl`             |
| `trellis-implement` | Write code                                  | No `git commit` allowed                 | `implement.jsonl` + `prd.md` |
| `trellis-check`     | Verify and self-fix                         | Writes only to fix; owns its retry loop | `check.jsonl` + `prd.md`     |

> The 0.4 era six sub-agents (`dispatch` / `plan` / `debug` / `research` / `implement` / `check`) and the SubagentStop-driven Ralph Loop were removed in 0.5: dispatch / plan / debug got replaced by skill routing; Ralph Loop was replaced by an internal retry loop inside `trellis-check`.

**Typical task flow in the main session**:

```
trellis-brainstorm skill
  │  Clarify scope, draft prd.md, create task dir + JSONL config
  │  Spawn trellis-research sub-agent in parallel if research needed
  ▼
trellis-before-dev skill
  │  Pre-read relevant specs before coding
  ▼
Spawn trellis-implement sub-agent
  │  Hook injects implement.jsonl context
  │  Sub-agent writes code; no git commit
  ▼
Spawn trellis-check sub-agent
  │  Hook injects check.jsonl context
  │  Runs lint / typecheck / test, compares diff against quality checklist
  │  Self-fixes inside its internal retry loop
  ▼
[Human verification / real-world business test]
  │
  ▼
[Phase 3.4: human commits]
  │
  ▼
/trellis:finish-work
  │  Archives task + appends session entry to workspace journal
```

## Workspace System

Carries per-developer working context across sessions, isolated per developer.

```
.trellis/workspace/
├── index.md                # Global index (list of all developers)
└── {developer}/            # Per-developer
    ├── index.md            # Personal journal index (auto-maintained via @@@auto markers)
    └── journal-N.md        # Session journal (~2000 lines per file)
```

**Key files**:

| File                                            | Purpose                                         | gitignored       |
| ----------------------------------------------- | ----------------------------------------------- | ---------------- |
| `.trellis/.developer`                           | Developer identity (unique key)                 | yes (per-clone)  |
| `.trellis/.current-task`                        | Main session active-task pointer                | yes              |
| `.trellis/.runtime/sessions/<session-key>.json` | Session-scoped task pointer (multi-window safe) | yes              |
| `workspace/{name}/journal-*.md`                 | Session journal body                            | no (team-shared) |
| `workspace/{name}/index.md`                     | Personal index (auto-maintained)                | no               |

> Journal rotation: when a single file exceeds 2000 lines, a new `journal-N+1.md` starts; old journals stay readable. `/trellis:finish-work` calls `add_session.py` to write new entries.

## Task System

A task is the unit of work that the AI operates on. Each task is one directory carrying requirements, research, context lists, and metadata.

```
.trellis/tasks/{MM-DD-slug}/
├── task.json              # Metadata: status / phase / branch / assignee / subtasks
├── prd.md                 # Requirements + acceptance criteria + non-goals
├── info.md                # Technical design (optional)
├── implement.jsonl        # implement sub-agent context list
├── check.jsonl            # check sub-agent context list
├── research.jsonl         # research sub-agent context list (optional)
└── research/              # Research outputs (written by trellis-research)
    ├── overview.md
    └── *.md
```

**`task.json` key fields**:

| Field                  | Meaning                                                    |
| ---------------------- | ---------------------------------------------------------- |
| `id` / `slug`          | Task identity                                              |
| `status`               | `planning` / `in_progress` / `completed` (auto-maintained) |
| `phase`                | Current step (e.g. `2.1`, `3.4`)                           |
| `branch`               | Linked git branch                                          |
| `creator` / `assignee` | Creator / assigned developer                               |
| `parent` / `children`  | Parent / subtask tree                                      |
| `prd_path` / `pr_url`  | Linked PRD and PR                                          |

**Task lifecycle hooks** (`.trellis/config.yaml`):

```yaml
hooks:
  after_create:
    - command: "linear-issue create --title '{task.title}'"
  after_start:
    - command: 'linear-issue update --status in_progress'
  after_finish:
    - command: 'linear-issue update --status done'
  after_archive:
    - command: 'linear-issue close'
```

Four trigger points; each can run an arbitrary shell command for syncing with Linear / Jira / internal systems. Hook failures don't block the main flow.

**`task.py` 16 subcommands** (grouped by purpose):

| Group     | Commands                                                   |
| --------- | ---------------------------------------------------------- |
| Lifecycle | `create` / `start` / `finish` / `archive` / `restore`      |
| Context   | `add-context` / `remove-context` / `list-context`          |
| Metadata  | `set-status` / `set-phase` / `set-assignee` / `set-branch` |
| Hierarchy | `add-subtask` / `remove-subtask`                           |
| PR        | `set-pr` / `unset-pr`                                      |

Run `python3 ./.trellis/scripts/task.py --help` for full signatures.

## Multi-task Parallelism

Trellis treats "multiple tasks in flight" as a default scenario, not an afterthought. Three layers of isolation stack:

### 1. Per-task isolation

Every task has its own directory `.trellis/tasks/<MM-DD-slug>/`. The PRD, `research/`, `implement.jsonl`, and `check.jsonl` are scoped to that directory. Sub-agents only read the JSONL of the **active** task, so cross-task spec injection is bounded automatically — task A's dead-end findings can't contaminate task B's implementation.

### 2. Per-session isolation (multi-window safe)

```
.trellis/.runtime/sessions/
├── 7f3a2b1c4d5e.json      # window 1 active task: 04-26-add-audit-log
├── 8e1d9c2a6f3b.json      # window 2 active task: 04-27-monorepo-migrate
└── ...
```

When you have multiple IDE windows / terminal sessions running Trellis at the same time, each session gets its own `<session-key>.json` tracking its own active task. The SessionStart hook assigns a key on session creation (based on PID + timestamp); subsequent hooks, `/trellis:continue`, and sub-agent injection all read the pointer for their own key. **Switching task X in window A does not interrupt task Y in window B.**

`.trellis/.current-task` is a global fallback pointer for command-line `task.py` users. Session-scoped pointers take precedence.

### 3. Subtask tree

Complex tasks can be decomposed:

```
.trellis/tasks/
├── 04-26-payment-revamp/           # parent
│   ├── task.json                   # children: [04-26-payment-stripe, 04-26-payment-alipay]
│   └── prd.md
├── 04-26-payment-stripe/           # child
│   ├── task.json                   # parent: 04-26-payment-revamp
│   └── prd.md
└── 04-26-payment-alipay/           # child
    └── ...
```

Parent tasks don't write code directly — they carry requirements and shared research. Each child runs its own complete Phase 1-3 loop. `task.py add-subtask <parent> <child>` establishes the relationship; brainstorm phase usually auto-decomposes complex tasks at creation time.

### 4. Cross-task knowledge flow

Parallel tasks don't share context, but there are three flow channels (sorted by durability):

| Channel                        | Purpose                                                    | Flow scope                                                 |
| ------------------------------ | ---------------------------------------------------------- | ---------------------------------------------------------- |
| `.trellis/spec/`               | Lift durable findings via `/trellis:update-spec`           | Auto-visible to all future tasks (whenever JSONL lists it) |
| `workspace/<dev>/journal-*.md` | Personal session continuity                                | Same developer's subsequent sessions                       |
| Task PRD reference             | Task B's `prd.md` explicitly links task A's PRD / research | Pulled in only during brainstorm of the referencing task   |

**What does NOT auto-flow**: task A's `research/` is not auto-injected into task B. Durable findings → `update-spec` lifts them to spec. Task-specific findings → leave them in the source task and reference explicitly from the new task.

### About the `/parallel` command

0.4-era Trellis shipped `/parallel` plus `.trellis/scripts/multi_agent/` (Git worktree orchestration, multiple agents each running in their own worktree). 0.5 removed it — modern agent CLIs / IDEs (Claude Code, Cursor, OpenCode, etc.) have native worktree support, so Trellis doesn't ship its own anymore. To run multiple agents in parallel: use the platform's native worktree entry to create separate worktrees, then run an independent Trellis session in each one.

## Spec System

The source of truth for team conventions, injected into sub-agents on demand. Supports both single-repo and monorepo layouts.

```
# Single repo
.trellis/spec/
├── frontend/                  # Frontend conventions
│   ├── index.md               # ← One-line catalogue of files in this directory
│   ├── components.md
│   └── state-management.md
├── backend/                   # Backend conventions
│   ├── index.md
│   ├── error-handling.md
│   └── database.md
└── guides/                    # Cross-cutting thinking guides
    ├── index.md
    └── cross-layer-thinking-guide.md

# Monorepo (per-package)
.trellis/spec/
├── <package-name>/
│   ├── backend/
│   ├── frontend/
│   └── unit-test/
└── guides/                    # Shared guides
```

**`index.md` is the discovery surface**: each spec layer's `index.md` is a one-line catalogue + short reason. The brainstorm/research phase reads only the `index.md` (paths + reasons), then writes the relevant entries into `implement.jsonl` / `check.jsonl`. Body files load only when the JSONL lists them. This is the key to on-demand injection — a 500K-line repo with 50 spec files but 4 loaded per task feels like a 4-file project to the AI.

**Protected paths**: `trellis update` will never overwrite `.trellis/spec/frontend/`, `.trellis/spec/backend/`, or `.trellis/spec/<package>/`. Template updates only adjust scaffolding (e.g., new `guides/` content from a new release).

## Hooks

On hook-capable platforms, Trellis registers these hooks:

| Hook event                                     | When it fires                       | Trellis behavior                                                                        |
| ---------------------------------------------- | ----------------------------------- | --------------------------------------------------------------------------------------- |
| `SessionStart`                                 | New session opens (startup matcher) | Inject workflow + developer + git status + active tasks + spec index (~16.7 KB payload) |
| `UserPromptSubmit`                             | Every user message                  | Inject `<workflow-state>` breadcrumb (~200 bytes, keyed by current task `status`)       |
| `PreToolUse:Agent` / Bun plugin `chat.message` | Before sub-agent starts             | Inject specs / PRD via JSONL (hook-push platforms only)                                 |

**Underlying Python scripts** (the actual hook implementations):

| Script                       | Bound event                   |
| ---------------------------- | ----------------------------- |
| `session-start.py`           | SessionStart                  |
| `inject-workflow-state.py`   | UserPromptSubmit              |
| `inject-subagent-context.py` | PreToolUse:Agent / Bun plugin |

> The 0.4-era `ralph-loop.py` (SubagentStop hook implementing the check-agent retry loop) was removed in 0.5 — the check sub-agent owns its internal retry loop now.

Hook installation paths differ by platform (`.claude/hooks/` / `.cursor/hooks/` / `.opencode/plugin/` / `.codex/hooks/` etc.) but they all share the same Python implementations under `shared-hooks/`; only the output adapter differs. Forking the workflow means editing `workflow.md`; you almost never have to touch Python.
