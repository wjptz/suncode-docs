## Appendix B: Command & Skill Cheat Sheet

Suncode ships only a small session-boundary command surface. Everything else is an auto-trigger skill or a sub-agent. This is intentional: commands are for explicit user boundaries; the rest of the workflow runs itself.

### Slash Commands (per platform)

`finish-work` and `continue` are the normal user-invoked commands. `start` is only user-facing on platforms that do not auto-inject session context.

| Platform                                                           | Start                                                                     | Finish                       | Continue                  | Delivery form                          |
| ------------------------------------------------------------------ | ------------------------------------------------------------------------- | ---------------------------- | ------------------------- | -------------------------------------- |
| Claude Code / OpenCode / Engineer / Gemini CLI / CodeBuddy / Droid | automatic SessionStart hook                                               | `/suncode:finish-work`       | `/suncode:continue`       | Slash command                          |
| Cursor / Pi Agent / Qoder                                          | automatic SessionStart hook or extension                                  | `/suncode-finish-work`       | `/suncode-continue`       | Native command / prompt                |
| Kiro                                                               | `@suncode:start`                                                          | `@suncode:finish-work`       | `@suncode:continue`       | Skill via `@` picker                   |
| Kilo                                                               | `/start.md`                                                               | `/finish-work.md`            | `/continue.md`            | Workflow file (`.kilocode/workflows/`) |
| Antigravity / Devin                                                | workflow picker → `start` / `finish-work` / `continue`                    |                              |                           | Workflow file                          |
| GitHub Copilot                                                     | Run Prompt → `suncode-start` / `suncode-finish-work` / `suncode-continue` |                              |                           | Prompt file (`.github/prompts/`)       |
| Codex                                                              | automatic `AGENTS.md` prelude; optional UserPromptSubmit hook             | skill / prompt entry         | skill / prompt entry      | Skills + hooks                         |
| Oh My Pi / Grok Build                                              | platform-specific Suncode command                                         | Suncode command              | Suncode command           | Native command files                   |
| Kimi Code                                                          | `/skill:suncode-start`                                                    | `/skill:suncode-finish-work` | `/skill:suncode-continue` | Private Kimi skills                    |

### Auto-trigger Skills

These are matched by the platform based on user intent: no explicit invocation required. Trigger them manually with your platform's skill invocation syntax if needed.

| Skill                 | Triggers when…                                        | What it does                                        |
| --------------------- | ----------------------------------------------------- | --------------------------------------------------- |
| `suncode-brainstorm`  | User describes a feature / bug / ambiguous request    | Produces task + `prd.md`, spawns research as needed |
| `suncode-before-dev`  | Task is in_progress and the AI is about to write code | Reads relevant spec files for the package           |
| `suncode-check`       | Implementation phase finished                         | Diff review, lint / typecheck / test, self-fix loop |
| `suncode-update-spec` | A learning / decision / gotcha is worth capturing     | Adds an entry to the right spec file                |
| `suncode-break-loop`  | A tricky bug was just resolved                        | 5-dimension root-cause + prevention analysis        |

### Sub-agents

Spawned by the main session via the platform's sub-agent / Task primitive.

| Sub-agent           | Role                  | Restriction            |
| ------------------- | --------------------- | ---------------------- |
| `suncode-research`  | Codebase / doc search | Read-only              |
| `suncode-implement` | Coding                | No `git commit`        |
| `suncode-check`     | Verify + self-fix     | Has its own retry loop |

Available as sub-agent workflows on Claude Code, Cursor, OpenCode, Engineer, Codex, Kiro, Gemini CLI, Qoder, CodeBuddy, Copilot, Droid, Pi Agent, Oh My Pi, Reasonix, ZCode, Trae, Grok Build, and Kimi Code. Kilo, Antigravity, and Devin run the same work inline in the main session. Codex uses native `SubagentStart` context when dispatch is enabled and retains a pull fallback; Grok and Kimi are pull-based.

### Global CLI Commands

| Command                 | Purpose                                                           | Usage                                                              |
| ----------------------- | ----------------------------------------------------------------- | ------------------------------------------------------------------ |
| `suncode upgrade`       | Upgrade the globally installed Suncode CLI package                | `suncode upgrade [--tag <tag>]`                                    |
| `suncode update`        | Sync the current project's Suncode files to this CLI              | `suncode update [--dry-run] [--migrate] [...]`                     |
| `suncode platforms`     | Report configured Suncode platforms                               | `suncode platforms [--json]`                                       |
| `suncode channel spawn` | Register a collaboration worker; Codex accepts a sandbox override | `suncode channel spawn <name> --provider codex [--sandbox <mode>]` |

`suncode platforms --json` prints a stable
`{"platforms":[{"id":"codex","displayName":"Codex","configDir":".codex"}]}`
document. The empty
result is `{"platforms":[]}` with exit 0. Codex channel sandbox modes are
`read-only`, `workspace-write` (default), and `danger-full-access`; other
providers ignore the option.

### Hub CLI Commands

Hub commands are optional. They are active only when `.suncode/config.yaml` contains `hub.enabled: true` and the project is signed in with `suncode hub login`.

| Command                         | Purpose                                                                   | Usage                                                              |
| ------------------------------- | ------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| `suncode hub init`              | Write Hub project config and user-level default Hub URL                   | `suncode hub init [--no-auto-pull-spec] [--yes] [...]`             |
| `suncode hub login`             | Sign in to the configured Hub service                                     | `suncode hub login [--api-base-url <url>] [--email <email>]`       |
| `suncode hub logout`            | Remove the local Hub session for a service                                | `suncode hub logout [--api-base-url <url>]`                        |
| `suncode hub state`             | Show config, login, service, work, and current-task state                 | `suncode hub state [--json] [--prompt] [--hook]`                   |
| `suncode hub status`            | Show whether Hub is enabled for this project                              | `suncode hub status`                                               |
| `suncode hub pull`              | List requirements assigned to the current developer                       | `suncode hub pull`                                                 |
| `suncode hub intake`            | Claim a Hub requirement into a local Suncode task                         | `suncode hub intake --auto` or `--requirement <id>`                |
| `suncode hub pull-spec`         | Pull the authoritative project spec bundle from Hub                       | `suncode hub pull-spec [--json]`                                   |
| `suncode hub sync`              | Pull latest requirement details and requirement changes                   | `suncode hub sync --task current [--cursor <cursor>]`              |
| `suncode hub plan-ready`        | Submit plan artifacts and structured subtasks                             | `suncode hub plan-ready --task current [--debug]`                  |
| `suncode hub review`            | Run a post-implementation Hub code review round                           | `suncode hub review --task current [--module <path>]`              |
| `suncode hub latest-review`     | Read the latest local Hub review result                                   | `suncode hub latest-review --task current`                         |
| `suncode hub pull-review`       | Pull Hub-side plan/start review comments                                  | `suncode hub pull-review --task current [--cursor <cursor>]`       |
| `suncode hub finish`            | Submit final spec and completion artifacts                                | `suncode hub finish --task current [--file <path>]`                |
| `suncode hub submit-plan`       | Upload current task planning artifacts                                    | `suncode hub submit-plan --task current [--force]`                 |
| `suncode hub submit-subtasks`   | Upload structured subtasks derived from `subtasks.json` or `implement.md` | `suncode hub submit-subtasks --task current [--force]`             |
| `suncode hub submit-spec`       | Upload project-level spec artifacts                                       | `suncode hub submit-spec --task current [--file <path>] [--force]` |
| `suncode hub submit-completion` | Upload completion artifacts                                               | `suncode hub submit-completion --task current [--force]`           |
| `suncode hub create-task`       | Create or bind the remote Hub task for a local task                       | `suncode hub create-task --task current [--best-effort]`           |
| `suncode hub preflight-start`   | Ask Hub whether the task can start development                            | `suncode hub preflight-start --task current`                       |
| `suncode hub mark-started`      | Mark a bound Hub task as started                                          | `suncode hub mark-started --task current [--status in_progress]`   |
| `suncode hub sync-pending`      | Retry queued best-effort Hub lifecycle sync failures                      | `suncode hub sync-pending`                                         |
| `suncode hub knowledge`         | Search the current project's Hub knowledge base                           | `suncode hub knowledge "<query>" [--top-k <n>]`                    |
| `suncode hub skill-push`        | Upload a local `.agents/skills/<name>/` package                           | `suncode hub skill-push <skill-name>`                              |
| `suncode hub skill-pull`        | Download a Hub skill package into `.agents/skills/`                       | `suncode hub skill-pull <skill-name>`                              |
| `suncode hub agent-push`        | Upload a local `.suncode/agents/<name>.md` package                        | `suncode hub agent-push <agent-name>`                              |
| `suncode hub agent-pull`        | Download a Hub agent package into `.suncode/agents/`                      | `suncode hub agent-pull <agent-name>`                              |

`suncode hub spec-deletions list|keep|discard` manages local spec candidates that Hub deleted but Suncode preserved for review. `suncode hub download-document` is a lower-level helper for signed Hub document payloads.

### `task.py` Subcommands

| Subcommand        | Purpose                                                                                           | Usage                                                              |
| ----------------- | ------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| `create`          | Create task (also seeds `implement.jsonl` / `check.jsonl` when a sub-agent platform is installed) | `task.py create "title" [--slug name] [-a assignee] [-p priority]` |
| `add-context`     | Add context entry (primary way to populate jsonl after `create`)                                  | `task.py add-context "$DIR" <target> "<path>" "<reason>"`          |
| `validate`        | Validate JSONL                                                                                    | `task.py validate "$DIR"`                                          |
| `list-context`    | View all entries                                                                                  | `task.py list-context "$DIR"`                                      |
| `start`           | Set as current task for this AI session/window                                                    | `task.py start "$DIR"`                                             |
| `current`         | Show the current task for this AI session/window                                                  | `task.py current [--source] [--json]`                              |
| `finish`          | Clear current task for this AI session/window                                                     | `task.py finish`                                                   |
| `set-branch`      | Set branch name                                                                                   | `task.py set-branch "$DIR" "feature/xxx"`                          |
| `set-base-branch` | Set PR target branch                                                                              | `task.py set-base-branch "$DIR" "main"`                            |
| `set-scope`       | Set scope                                                                                         | `task.py set-scope "$DIR" "auth"`                                  |
| `add-subtask`     | Link child to parent                                                                              | `task.py add-subtask <parent-dir> <child-dir>`                     |
| `remove-subtask`  | Unlink child from parent                                                                          | `task.py remove-subtask <parent-dir> <child-dir>`                  |
| `archive`         | Archive task                                                                                      | `task.py archive <name>`                                           |
| `list`            | List active tasks                                                                                 | `task.py list [--mine] [--status <s>] [--json]`                    |
| `list-archive`    | List archived tasks                                                                               | `task.py list-archive [YYYY-MM]`                                   |

`task.py create` also accepts Hub metadata options such as `--hub-project-id`, `--hub-developer-id`, `--hub-requirement-id`, `--hub-requirement-revision`, `--hub-task-role`, `--hub-parent-local-task-id`, and `--hub-parent-remote-task-id`. Normal Hub intake should still use `suncode hub intake`; these flags are for advanced binding and tests.

`task.py current --json` emits `{current_task, source, stale}` and exits 1
when `current_task` is `null`. `task.py list --json` emits `{tasks:[...]}`;
each item includes both stored `status` and derived `display_status`, so a
parent can display as `active` while its persisted status remains `planning`.

### Python Scripts

```bash
# Context
./.suncode/scripts/get_context.py                 # Full context
./.suncode/scripts/get_context.py --json          # JSON
./.suncode/scripts/get_context.py --mode packages # Per-package spec layers (monorepo)
./.suncode/scripts/get_context.py --mode record   # For /suncode:finish-work

# Session
./.suncode/scripts/add_session.py --title "..." --commit "..." --summary "..."

# Spec bootstrap (first-time)
./.suncode/scripts/create_bootstrap.py
```
