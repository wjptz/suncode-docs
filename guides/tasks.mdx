## What tasks are for

Tasks are work items with their own context. Each task knows:

- What you're building (PRD)
- Which files are relevant
- What branch to work on
- Current status

When you switch tasks, the AI switches context. No confusion about which feature you're working on.

## Task structure

Tasks live in `.trellis/tasks/`. Each task is a directory:

```
.trellis/tasks/01-31-user-auth-taosu/
├── task.json           # Metadata
├── prd.md              # Requirements
├── implement.jsonl     # Context for implement agent
├── check.jsonl         # Context for check agent
└── debug.jsonl         # Context for debug agent
```

The directory name follows the pattern: `MM-DD-slug-assignee`.

## Creating a task

Create the task directory manually:

```bash
mkdir -p .trellis/tasks/01-31-user-auth-taosu
```

Then add the required files inside.

### task.json

```json
{
  "id": "01-31-user-auth-taosu",
  "title": "Add user authentication",
  "status": "in_progress",
  "assignee": "taosu",
  "branch": "feature/user-auth",
  "created": "2026-01-31",
  "phases": ["implement", "check"]
}
```

### prd.md

Write what you're building. Be specific about requirements and acceptance criteria.

```markdown
# User Authentication

## Goal

Users can sign up and log in with email/password.

## Requirements

- Sign up with email, password, display name
- Email verification required before login
- Login returns JWT token
- Token expires after 7 days
- Password reset via email

## API Endpoints

- POST /auth/signup
- POST /auth/login
- POST /auth/verify-email
- POST /auth/forgot-password
- POST /auth/reset-password

## Out of scope

- Social login (Google, GitHub)
- Two-factor authentication
```

### Context files (JSONL)

These tell agents which files to read. One file path per line:

```jsonl
{"type": "spec", "path": ".trellis/spec/backend/index.md"}
{"type": "spec", "path": ".trellis/spec/backend/api.md"}
{"type": "file", "path": "src/routes/auth.ts"}
{"type": "file", "path": "src/services/user.ts"}
{"type": "file", "path": "src/db/models/user.ts"}
```

Different agents can have different context. The `implement` agent might need different files than the `check` agent.

## Working on a task

Set the current task by writing the task ID to `.trellis/.current-task`:

```bash
echo "01-31-user-auth-taosu" > .trellis/.current-task
```

Hooks use this file to inject the right context. Now when you start a Claude Code session, it knows:

- Read the PRD from this task
- Load specs listed in `implement.jsonl`
- Work on the `feature/user-auth` branch

## Task status

Update status by editing the `status` field in `task.json`:

```json
{
  "status": "in_progress"
}
```

Common statuses: `pending`, `in_progress`, `review`, `done`.

## Multiple tasks

You can have multiple tasks in progress. Only one is "current" at a time. Switch by updating `.trellis/.current-task`:

```bash
echo "other-task-id" > .trellis/.current-task
```

## Task workflow

Typical flow:

1. Create task with requirements
2. Set up context files (what specs and code are relevant)
3. Start the task
4. Work in Claude Code sessions
5. Mark complete when done

The task system doesn't enforce a workflow. It just keeps context organized so the AI knows what to focus on.
