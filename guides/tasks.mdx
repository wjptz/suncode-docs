## What tasks are for

Tasks are durable work items with their own planning artifacts and context.
They are for work that should survive chat compaction, tool restarts, and
handoff across agents or platforms.

Simple conversation and small inline edits do not automatically create Trellis
tasks. The AI first classifies the current turn and asks for task-creation
consent before writing anything under `.trellis/tasks/`.

## When a task is needed

Trellis uses three practical levels:

| Level               | Use for                                                                                   | Consent rule                                                                                                                                              |
| ------------------- | ----------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Simple conversation | Questions, explanations, lookups, or discussion with no repo change                       | No task by default. If the AI thinks a task might help, it asks only whether this turn should create a Trellis task.                                      |
| Inline small task   | A contained edit that can be understood and verified in the current turn                  | The AI asks only whether this turn should create a Trellis task. If you say no, it skips Trellis for this turn and may proceed inline.                    |
| Full Trellis task   | Multi-file work, workflow/spec/platform changes, generated templates, or durable planning | The AI explains why a task is useful and asks whether it may create a Trellis task and enter planning. If you say no, it should not do broad inline work. |

Large programs of work are split into related Trellis tasks instead of being
forced into one giant task directory.

## Task structure

Tasks live in `.trellis/tasks/`. Each task is a directory:

```text
.trellis/tasks/05-10-task-artifacts-taosu/
├── task.json          # Metadata and status
├── prd.md             # Requirements and acceptance criteria
├── design.md          # Optional: technical design for complex tasks
├── implement.md       # Optional: implementation plan for complex tasks
├── implement.jsonl    # Spec/research manifest for implementation
├── check.jsonl        # Spec/research manifest for review
└── research/          # Optional durable investigation notes
```

The directory name follows the pattern `MM-DD-slug-assignee` when an assignee is
known.

Lightweight tasks can have only `prd.md`. Complex tasks must have `prd.md`,
`design.md`, and `implement.md` before `task.py start`.

## Creating a task

Use the task script:

```bash
python3 .trellis/scripts/task.py create "user-auth" --assignee taosu
```

This creates the task directory, writes `task.json`, and creates the default
`prd.md` template. Sub-agent-capable setups may also seed `implement.jsonl` and
`check.jsonl`.

`task.py create` does not create `design.md` or `implement.md`. The AI writes
those files during planning when the task is complex enough to need them.

## Planning artifacts

### prd.md

`prd.md` records the product requirement and review target. It should say what
must change, what constraints apply, and how the finished work will be accepted.
It should not carry the technical design or execution checklist.

```markdown
# User Authentication

## Goal

Users can sign up and log in with email/password.

## Requirements

- Sign up with email, password, and display name
- Email verification required before login
- Login returns a JWT token
- Token expires after 7 days
- Password reset works through email

## Acceptance Criteria

- Signup rejects duplicate emails
- Unverified accounts cannot log in
- Password reset tokens expire

## Out of Scope

- Social login
- Two-factor authentication
```

### design.md

`design.md` is for complex tasks. It explains the technical shape:

- affected modules and boundaries
- data flow and contracts
- hook, workflow, or platform integration points
- compatibility and migration concerns
- tradeoffs and rollback shape

### implement.md

`implement.md` is the execution plan for complex tasks. It should contain:

- ordered implementation checklist
- validation commands
- review gates
- rollback points
- known sequencing constraints

It does not replace `implement.jsonl`. The markdown file is the plan; the JSONL
file is the spec/research manifest used for context loading.

## Context manifests

`implement.jsonl` and `check.jsonl` tell agents which stable context files to
read. They should list specs and research, not source files that are about to be
edited.

```jsonl
{"file": ".trellis/spec/cli/backend/index.md", "reason": "CLI backend contract"}
{"file": ".trellis/tasks/05-10-example/research/platforms.md", "reason": "Platform behavior research"}
```

Use `implement.jsonl` for implementation-writing context and `check.jsonl` for
review and verification context. Inline-only work can rely on direct reading
instead of curated JSONL when there is no sub-agent boundary.

## Working on a task

Set the current task only after the planning artifacts are ready:

```bash
python3 .trellis/scripts/task.py start 05-10-task-artifacts-taosu
```

For a lightweight task, `prd.md` can be enough. For a complex task, starting the
task requires the full planning set:

- `prd.md`
- `design.md`
- `implement.md`

Hooks and skills then load the active task and keep the AI aligned with the
current workflow phase.

## Managing tasks

List all tasks:

```bash
python3 .trellis/scripts/task.py list
```

Clear current task when done:

```bash
python3 .trellis/scripts/task.py finish
```

Archive completed task:

```bash
python3 .trellis/scripts/task.py archive 05-10-task-artifacts-taosu
```

## Typical workflow

1. Classify the turn and get task-creation consent when a task might be needed.
2. Create the task with `task.py create`.
3. Fill `prd.md` with requirements and acceptance criteria.
4. For complex tasks, add `design.md` and `implement.md`.
5. Add spec/research manifests when sub-agent or review context needs them.
6. Start the task with `task.py start`.
7. Implement, check, and update specs as needed.
8. Commit the work, then run finish-work and archive bookkeeping.

The task system keeps durable planning and context in files so the next AI
session can continue without depending on the previous chat transcript.
