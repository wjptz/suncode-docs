---
title: 'Parallel Sessions'
description: 'Run multiple AI sessions without sharing active task state'
---

## Current Model

Trellis 0.5 removed the old Multi-Agent Pipeline, `/trellis:parallel`, `worktree.yaml`, and the six-agent plan/dispatch/debug pipeline. Use your AI tool's native sessions and Git worktrees instead.

The important Trellis change is that the active task is now **session-scoped**:

```
.trellis/.runtime/sessions/<session-key>.json
```

That means window A can work on task A while window B works on task B. Starting a task in one AI session does not overwrite another session's active task.

## Recommended Flow

1. Create one Trellis task per independent work item.
2. Open one AI session/window per task.
3. In each session, run `task.py start "$TASK_DIR"` for that task.
4. Let each session follow the normal Plan → Execute → Finish workflow.
5. Review and merge branches using your normal Git process.

For isolated code changes, create Git worktrees yourself:

```bash
git worktree add ../feature-login -b feature/login
git worktree add ../feature-settings -b feature/settings
```

Then run a separate AI session inside each worktree.

## When This Works

Good fit:

- Independent features
- Parallel bug fixes
- Separate packages or files
- Experiments where each branch can be reviewed independently

Poor fit:

- Tasks that touch the same files
- Tasks that depend on each other's unmerged changes
- Work that needs tight user feedback in one conversation

## Context Injection

Hook/extension-backed platforms inject the active task context for that session. Pull-based platforms read the task JSONL files from their sub-agent prelude. In both cases, the session key decides which task is active; there is no global active-task pointer.
