---
title: 'How It Works'
---

Walk through one real request from start to finish. By the end you'll know which hooks, skills, sub-agents, and files get strung together when you send a prompt, and how it ends at a commit.

> This is the user-facing flow. For component-level details, extension points, and forking guidance, see [Architecture Overview](/advanced/architecture).

## 1. You open a session

Launch Claude Code, Cursor, Codex, or any supported AI coding platform inside a project that has `.trellis/`.

Hook-capable platforms fire the SessionStart hook automatically. It pushes the following into the AI's initial context:

- Current `git status`, recent commits, current branch
- The `.trellis/spec/` tree (the team conventions library)
- The session's active task pointer, if any
- The `Phase Index` plus full step walkthrough from `.trellis/workflow.md`

Different platforms use different mechanisms:

| Platform                                            | Injection mechanism                                   |
| --------------------------------------------------- | ----------------------------------------------------- |
| Claude Code, Cursor, OpenCode, CodeBuddy, Droid, Pi | SessionStart hook pushes context before the AI starts |
| Codex, Copilot, Gemini, Qoder                       | Sub-agents pull via prelude (hook optional)           |
| Kilo, Antigravity, Windsurf                         | No hook primitive; skills are loaded on demand        |

By the time your terminal opens, the AI already knows this is a Trellis project, what the active task is, and where the conventions live in `spec/`. You don't `cat workflow.md` yourself.

## 2. You send your first prompt

Every user message triggers the `UserPromptSubmit` hook. The hook reads `task.json.status`, looks up the matching `[workflow-state:STATUS]` block in `.trellis/workflow.md`, wraps it in `<workflow-state>...</workflow-state>`, and injects it into this turn.

The AI sees this breadcrumb on every single message. It tells the AI which phase you're in (`no_task` / `planning` / `in_progress` / `completed`), what to do next, which steps are `[required]` and can't be skipped, and which user phrases let the AI bypass the flow.

```text
<workflow-state>
Status: no_task
A — Direct answer: pure Q&A / lookup / chat...
B — Create a task: any implementation / change / build / refactor work...
C — Inline change: per-turn escape hatch for B...
</workflow-state>
```

The AI doesn't rely on remembering to read `workflow.md`. The hook re-feeds the current phase reminder every turn.

## 3. The AI picks a path

Reading your prompt and the breadcrumb, the AI picks one of three paths:

- **A — Direct answer**: pure Q&A, explanation, doc lookup, chat. Decision rule is "no file writes, single-turn answer, at most 2 repo files to read." Answer and move on.
- **B — Create a task**: you described work to be done — implement, change, build, refactor. However small it looks, this enters the Trellis flow.
- **C — Inline change**: you said the magic phrase ("just do it" / "skip trellis" / "do it inline" / etc). The AI bypasses the flow this turn. Per-turn only; next turn falls back to B.

The rest of the steps below trace the B path.

## 4. Create a task

The AI runs:

```bash
python3 ./.trellis/scripts/task.py create "<title>" --slug <name>
```

Three things happen: `.trellis/tasks/<MM-DD-name>/` is created, `task.json` is written with `status: "planning"`, and the current session's active-task pointer is auto-set to the new task. The third one means no separate `start` step is needed yet.

Next turn the breadcrumb switches to `[workflow-state:planning]` and tells the AI: load the `trellis-brainstorm` skill, walk through requirements with the user, write `prd.md`. Once that's done, curate `implement.jsonl` and `check.jsonl`. Then `task.py start`.

Brainstorm runs next. The AI loads the `trellis-brainstorm` skill and runs Q&A with you. One question at a time; every answer is written into `prd.md` immediately. When the AI needs to research a third-party library or compare cross-platform options, it dispatches a `trellis-research` sub-agent. The sub-agent persists findings to `{task}/research/<topic>.md` so the main conversation stays clean.

With `prd.md` done, the AI moves to Phase 1.3 and curates two JSONL files:

- `implement.jsonl`: spec and research files the implement sub-agent should read while writing code
- `check.jsonl`: spec files the check sub-agent should read while verifying

Only spec paths and research paths go in. Source files do not — sub-agents read those at execution time.

Last, activate the task:

```bash
python3 ./.trellis/scripts/task.py start <task-dir>
```

`status` flips from `planning` to `in_progress`, the breadcrumb switches to `[workflow-state:in_progress]`, implementation begins.

## 5. Sub-agent implementation

The AI dispatches the `trellis-implement` sub-agent. The point: instead of having the main agent write code, the work is delegated.

Hooks auto-inject the sub-agent's initial context with `prd.md`, every spec file listed in `implement.jsonl`, and `info.md` if present. Three injection paths exist depending on platform:

| Type         | Platforms                                           | Mechanism                                                                               |
| ------------ | --------------------------------------------------- | --------------------------------------------------------------------------------------- |
| Push hook    | Claude Code, Cursor, OpenCode, CodeBuddy, Droid, Pi | `inject-subagent-context.py` hook pushes before sub-agent starts                        |
| Pull prelude | Codex, Copilot, Gemini, Qoder                       | Sub-agent definition embeds a prelude that `Read`s the files itself                     |
| Skill inline | Kilo, Antigravity, Windsurf                         | No sub-agent primitive — main session loads `trellis-before-dev` skill and reads inline |

The sub-agent writes the code, runs `lint` and `type-check`, and exits. It does not commit. The commit boundary stays with humans.

## 6. Automatic verification

The AI dispatches `trellis-check`. Same injection mechanism, this time wired to `check.jsonl`.

The check sub-agent does four things: reviews every change against the listed specs, runs `lint` / `type-check` / tests, runs cross-layer consistency checks when changes span multiple layers, and self-fixes any findings instead of just reporting them.

Pass advances to step 7. Fail reports findings, returns to step 5, re-checks afterward.

## 7. Distill and commit

The AI loads `trellis-update-spec` and asks itself: did this task produce anything worth writing back to `spec/`? Examples include a pitfall hit during implementation, a new pattern that emerged, a technical decision made — these go into `spec/<package>/<layer>/*.md` so future tasks' sub-agents pick them up automatically.

Then the main agent drives the commit (Phase 3.4, required). It states the commit plan in user-facing text, waits for your confirmation, runs `git commit`. It doesn't push. Push is yours.

Finally the AI prompts you to run `/trellis:finish-work`. That archives the task into `tasks/archive/YYYY-MM/` and writes a session journal to `.trellis/workspace/<dev>/journal-N.md`.

## Knowledge lives in files

After the full flow runs, everything is persisted:

- Your PRD, research, info → `tasks/<task>/`
- Distilled conventions → `spec/`
- Personal decision log → `workspace/<dev>/journal-N.md`
- Archived tasks → `tasks/archive/YYYY-MM/<task>/`

When the next session opens, the SessionStart hook indexes all of this into the AI's initial context. The whole team's sub-agents read it next time, not just the developer who wrote it.

## Why it works this way

Hooks push specs in before the AI sees your prompt — the AI doesn't have to remember to read them. Three-tier sub-agents (research, implement, check) keep each agent's scope narrow and constrain each other through clear handoffs. JSONL config injects only the specs each task actually uses, so context stays small. The AI doesn't commit; that's the PR review gate, and humans hold it. Conventions distilled back to `spec/` get read by everyone's sub-agent next time, not just yours.
