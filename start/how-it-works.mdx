title: 'How It Works'
---

Walk through one real request end-to-end and see what Trellis actually does at every step. After this you'll know exactly which hooks, skills, sub-agents, and files get strung together when you send a prompt — and how the work flows from your message to a final commit.

> This page is the user-facing flow. For component-level implementation details, extension points, and forking guidance, see [Architecture Overview](/advanced/architecture).

## Step 1: You open a session

You launch Claude Code / Cursor / Codex / any supported AI coding platform inside a project that has `.trellis/`.

**The SessionStart hook fires automatically** (on hook-capable platforms) and pushes the following into the AI's initial context:

- Current `git status`, recent commits, current branch
- `.trellis/spec/` index (the directory tree of the team conventions library)
- The session's active task pointer (if any)
- The `Phase Index` and full step walkthrough from `.trellis/workflow.md`

Different platforms use different mechanisms:

| Platform                                                 | Injection mechanism                               |
| -------------------------------------------------------- | ------------------------------------------------- |
| Claude Code / Cursor / OpenCode / CodeBuddy / Droid / Pi | SessionStart hook pushes context before AI starts |
| Codex / Copilot / Gemini / Qoder                         | Sub-agents pull via prelude (hook optional)       |
| Kilo / Antigravity / Windsurf                            | No hook primitive — skills are loaded on demand   |

**Outcome**: when you open the terminal, the AI already knows this is a Trellis project, what the active task is (if any), and where the conventions live in `spec/`. You don't have to `cat workflow.md` yourself.

## Step 2: You send your first prompt

Every user message triggers the `UserPromptSubmit` hook (on hook-capable platforms). The hook does one thing:

- Reads `task.json.status` (if there's an active task)
- Locates the matching `[workflow-state:STATUS]` block in `.trellis/workflow.md`
- Wraps the block content in `<workflow-state>...</workflow-state>` and injects it into this turn

This is the per-turn workflow breadcrumb the AI sees on every message. It tells the AI:

- Which phase you're in (no_task / planning / in_progress / completed)
- What to do next
- Which steps are `[required]` and can't be skipped
- Which user phrases let the AI bypass the flow

```text
<workflow-state>
Status: no_task
A — Direct answer: pure Q&A / lookup / chat...
B — Create a task: any implementation / change / build / refactor work...
C — Inline change: per-turn escape hatch for B...
</workflow-state>
```

**Outcome**: the AI doesn't need to "remember" to read `workflow.md`. The hook re-feeds the current phase reminder on every single turn.

## Step 3: The AI picks a path

After reading your prompt and the current breadcrumb, the AI picks one of three paths — **A / B / C**:

- **A — Direct answer**: pure Q&A, explanation, doc lookup, chat. Decision rule: no file writes + a single-turn answer + no more than 2 repo files to read. Just answer and move on.
- **B — Create a task**: you described implementation / change / build / refactor work. Regardless of how small it looks, this enters the Trellis flow.
- **C — Inline change**: you said the magic phrase ("just do it" / "skip trellis" / "do it inline" / etc) and the AI bypasses the flow this turn. Per-turn only — next turn falls back to B.

The rest of the steps below trace the **B path**.

## Step 4: Create a task

The AI runs:

```bash
python3 ./.trellis/scripts/task.py create "<title>" --slug <name>
```

Three things happen:

1. `.trellis/tasks/<MM-DD-name>/` directory is created
2. `task.json` is written with `status: "planning"`
3. **The current session's active-task pointer is set automatically** (no separate `start` step needed)

Next turn, the breadcrumb auto-switches to `[workflow-state:planning]`, which tells the AI:

> Now load the `trellis-brainstorm` skill, walk through requirements with the user, write `prd.md`. Once it's done, curate `implement.jsonl` / `check.jsonl`. Then `task.py start`.

**Brainstorm phase**: the AI loads the `trellis-brainstorm` skill, runs Q&A with you. One question at a time; every answer is written into `prd.md` immediately. When the AI needs to research a third-party library or compare cross-platform options, it dispatches a `trellis-research` sub-agent — the sub-agent persists findings to `{task}/research/<topic>.md`, keeping the main conversation clean.

**Phase 1.3 — JSONL curation**: with `prd.md` done, the AI curates two files:

- `implement.jsonl` — spec / research files the implement sub-agent should read while writing code
- `check.jsonl` — spec files the check sub-agent should read while verifying

Only spec paths and research paths go in. No source code files — sub-agents read those themselves at execution time.

**Activate the task**:

```bash
python3 ./.trellis/scripts/task.py start <task-dir>
```

`status: "planning" → "in_progress"`, the breadcrumb switches to `[workflow-state:in_progress]`, and implementation begins.

## Step 5: Sub-agent implementation

The AI dispatches the `trellis-implement` sub-agent. **The point**: instead of having the main agent write code, the work is delegated.

**Hooks auto-inject** the sub-agent's initial context:

- Full `prd.md`
- Every spec file listed in `implement.jsonl`
- `info.md` (if present)

Three injection paths, depending on the platform:

| Type         | Platforms                                                | Mechanism                                                                               |
| ------------ | -------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| Push hook    | Claude Code / Cursor / OpenCode / CodeBuddy / Droid / Pi | `inject-subagent-context.py` hook pushes before sub-agent starts                        |
| Pull prelude | Codex / Copilot / Gemini / Qoder                         | Sub-agent definition contains a prelude that `Read`s the files itself                   |
| Skill inline | Kilo / Antigravity / Windsurf                            | No sub-agent primitive — main session loads `trellis-before-dev` skill and reads inline |

The sub-agent writes the code, runs `lint` and `type-check`, and finishes. **It does not commit** — that boundary belongs to humans.

## Step 6: Automatic verification

The AI dispatches `trellis-check`. Same injection mechanism, this time wired to `check.jsonl`.

The check sub-agent does four things:

1. Reviews every change against the listed specs
2. Runs `lint` / `type-check` / tests
3. Cross-layer consistency check (when changes span multiple layers)
4. **Self-fixes any findings** — not just report

Pass → Step 7. Fail → reports findings, returns to Step 5, re-checks afterward.

## Step 7: Distill + commit

**3.3 update-spec**: the AI loads the `trellis-update-spec` skill and asks "did this task produce anything worth writing back to `spec/`?". Examples: a pitfall that was hit, a new pattern that emerged, a technical decision that was made — write it into `spec/<package>/<layer>/*.md`. Future tasks' sub-agents pick it up automatically.

**3.4 commit (required)**: the main agent drives the commit. It states the commit plan in user-facing text, waits for your confirmation, then runs `git commit`. **It does not push** — push is yours.

**3.5 wrap-up**: the AI prompts you to run `/trellis:finish-work`:

- `task.py archive` moves the task into `tasks/archive/YYYY-MM/`
- A session journal is written to `.trellis/workspace/<dev>/journal-N.md`

Task done.

## Knowledge lives in files, not conversations

After the full flow runs, **everything is persisted in files**:

- Your PRD / research / info → `tasks/<task>/`
- Distilled conventions → `spec/`
- Personal decision log → `workspace/<dev>/journal-N.md`
- Archived tasks → `tasks/archive/YYYY-MM/<task>/`

When the next session opens, the SessionStart hook indexes all of this into the AI's initial context. **The whole team's sub-agents read it next time** — not just yours.

## Why it's designed this way

Every step is here for a reason:

- **Hooks push before the AI sees your prompt** → the AI doesn't rely on "remembering" to read specs
- **Three-tier sub-agents (research / implement / check)** → each agent has bounded scope and clear responsibility, mutual constraint
- **JSONL config instead of dumping all specs** → context stays small, every task injects only what it actually uses
- **AI doesn't commit** → commit is the PR review gate; humans must decide
- **Distillation back to `spec/`** → solve once, benefit forever, compounding across the team
