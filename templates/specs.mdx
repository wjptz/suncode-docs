## Backend Templates

### Node.js + Express + TypeScript

<CodeGroup>
```markdown Backend Guidelines
# Backend Guidelines

## Project Structure

src/
├── routes/ # Express route handlers
├── services/ # Business logic
├── db/
│ ├── models/ # Database models
│ └── migrations/ # SQL migrations
├── lib/ # Shared utilities
└── types/ # TypeScript types

## Error Handling

All errors use AppError from src/lib/errors.ts.
API error response format: { "error": { "code": "...", "message": "..." } }
Never throw raw Error objects. Never expose stack traces in production.

## Database

- Table names: snake_case, plural (user_sessions)
- Column names: snake_case (created_at)
- Always include id, created_at, updated_at
- Foreign keys: {table_singular}\_id (user_id)
- Use the query builder from src/db/query.ts, not raw SQL.

## API Routes

Routes go in src/routes/. One file per resource.
Validate input with Zod. Schemas go in src/types/schemas.ts.

## Authentication

JWT tokens for auth. Token handling in src/lib/auth.ts.
Protected routes use requireAuth middleware.

````
</CodeGroup>

### Python + FastAPI

<CodeGroup>
```markdown FastAPI Guidelines
# Backend Guidelines

## Project Structure

app/
├── api/
│   └── routes/     # FastAPI route handlers
├── services/       # Business logic
├── db/
│   ├── models/     # SQLAlchemy models
│   └── migrations/ # Alembic migrations
├── core/           # Config, security, dependencies
└── schemas/        # Pydantic schemas

## Error Handling

Custom exceptions in app/core/exceptions.py.
Use the global exception handler. Don't catch and re-raise.

## Database

- Table names: snake_case, plural
- Use SQLAlchemy ORM, not raw SQL
- Models in app/db/models/
- Each model has id, created_at, updated_at

## API Routes

One router per resource in app/api/routes/.

## Validation

Use Pydantic schemas for all input/output.
````

</CodeGroup>

## Frontend Templates

### React + TypeScript

<CodeGroup>
```markdown Frontend Guidelines
# Frontend Guidelines

## Project Structure

src/
├── components/ # Reusable components
├── pages/ # Route pages
├── hooks/ # Custom hooks
├── stores/ # Zustand stores
├── lib/ # Utilities
├── types/ # TypeScript types
└── api/ # API client functions

## Components

Named exports only. No default exports.
Props interface defined above the component. No inline types.

## State Management

Use Zustand. Store files in src/stores/.

## API Calls

API functions in src/api/. Use the fetch wrapper from src/lib/fetch.ts.

## Styling

Tailwind CSS. No inline styles. No CSS modules.
For complex components, extract class names to variables.

````
</CodeGroup>

## Full-Stack Templates

### Next.js + tRPC + Prisma

<CodeGroup>
```markdown Full-Stack Guidelines
# Full-Stack Guidelines

## Project Structure

src/
├── app/         # Next.js app router
├── server/
│   ├── api/     # tRPC routers
│   └── db/      # Prisma client

Routers in src/server/api/routers/.

Models in prisma/schema.prisma.

Same as React guidelines. Use tRPC hooks for data fetching.
````

## Using These Templates

1. Copy the template to `.trellis/spec/backend/index.md` or `.trellis/spec/frontend/index.md`
2. Adjust paths to match your project
3. Add project-specific conventions
4. Remove sections that don't apply

Templates are starting points. Your spec should reference actual files in your codebase.
