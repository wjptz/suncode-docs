## Development commands

### Code review

`.claude/commands/dev/review.md`:

```markdown
Review the code I just wrote.

Check for:

1. **Type safety** - Are types correct? Any `any` that should be specific?
2. **Error handling** - Are errors caught and handled appropriately?
3. **Edge cases** - What inputs could break this?
4. **Performance** - Any obvious inefficiencies?
5. **Security** - Input validation, SQL injection, XSS?

For each issue found:

- Explain what's wrong
- Show how to fix it
- Rate severity: critical / warning / suggestion

If the code looks good, say so. Don't invent problems.
```

### Quick test

`.claude/commands/dev/test.md`:

```markdown
Write tests for the code I just wrote.

Use the project's test framework. Put tests next to the source file with `.test.ts` extension.

Cover:

- Happy path (normal usage)
- Edge cases (empty input, null, boundaries)
- Error cases (invalid input, failures)

Keep tests focused. One behavior per test. Use descriptive test names.

Run the tests after writing them. Fix any failures.
```

### Debug helper

`.claude/commands/dev/debug.md`:

```markdown
Help me debug this issue: $ARGUMENTS

Steps:

1. Understand the expected vs actual behavior
2. Identify where the bug might be
3. Add logging or breakpoints to narrow down
4. Find the root cause
5. Propose a fix

Don't guess. Trace the code path. Show your reasoning.
```

## Git commands

### Smart commit

`.claude/commands/git/commit.md`:

```markdown
Prepare a commit for my staged changes.

1. Run `git diff --cached` to see staged changes
2. Write a commit message following conventional commits:
   - `feat:` new feature
   - `fix:` bug fix
   - `docs:` documentation
   - `refactor:` code change that doesn't fix a bug or add a feature
   - `test:` adding or fixing tests
   - `chore:` maintenance

3. Keep the subject under 72 characters
4. Add a body if the change needs explanation

Show me the message before committing. Don't commit until I approve.
```

### PR description

`.claude/commands/git/pr.md`:

```markdown
Write a PR description for the current branch.

1. Run `git log main..HEAD --oneline` to see commits
2. Run `git diff main...HEAD --stat` to see changed files

Structure:

## Summary

What does this PR do? One paragraph.

## Changes

- Bullet list of specific changes

## Testing

How was this tested? What should reviewers check?

## Screenshots

(if UI changes)

Keep it concise. Reviewers are busy.
```

## Documentation commands

### API docs

`.claude/commands/docs/api.md`:

```markdown
Document this API endpoint: $ARGUMENTS

Include:

- HTTP method and path
- Request body (with types)
- Response format (success and error)
- Authentication requirements
- Example request and response

Format as markdown. Put in the appropriate docs location.
```

### README update

`.claude/commands/docs/readme.md`:

```markdown
Update the README based on recent changes.

Check:

1. Is the installation section current?
2. Are all commands documented?
3. Are environment variables listed?
4. Is the project structure accurate?

Don't rewrite sections that are fine. Only update what's outdated.
```

## Deployment commands

### Pre-deploy check

`.claude/commands/deploy/check.md`:

```markdown
Run pre-deployment checks:

1. **Tests**: `npm test`
2. **Types**: `npm run typecheck`
3. **Lint**: `npm run lint`
4. **Build**: `npm run build`

Run each command. Report results.

If any fail:

- Show the error
- Suggest a fix
- Don't proceed until fixed

If all pass, confirm ready to deploy.
```

### Environment check

`.claude/commands/deploy/env.md`:

```markdown
Check environment configuration for $ARGUMENTS environment.

1. List required environment variables from `.env.example`
2. Check which are set in the target environment
3. Flag any missing or obviously wrong values
4. Don't show actual secret values - just confirm they exist

Report:

- ✅ Variable set
- ❌ Variable missing
- ⚠️ Variable looks wrong (empty, placeholder value)
```

## Using these commands

1. Create the folder: `mkdir -p .claude/commands/dev`
2. Copy the command file
3. Use with `/dev:review`, `/git:commit`, etc.

Customize for your project:

- Change test framework commands
- Adjust commit message format
- Add project-specific checks
