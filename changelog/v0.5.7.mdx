Three Codex / Windows fixes. Sub-agent collab tools are now structurally disabled instead of prompt-layer-warned ([`#240`](https://github.com/mindfold-ai/Trellis/issues/240) follow-up, [`#241`](https://github.com/mindfold-ai/Trellis/issues/241)). The previously-dogfood-only "Load Trellis Context First" section is ported back to the canonical templates. Windows `trellis init` walks a Python command fallback chain instead of failing on the first miss ([`#236`](https://github.com/mindfold-ai/Trellis/issues/236)). No new migrations.

## Bug Fixes

### Codex sub-agent collab tools — structural disable ([`#240`](https://github.com/mindfold-ai/Trellis/issues/240) follow-up, [`#241`](https://github.com/mindfold-ai/Trellis/issues/241))

[v0.5.5](/changelog/v0.5.5) removed the `SessionStart` injection vector. [v0.5.6](/changelog/v0.5.6) added prompt-layer `fork_turns="none"` guidance to `AGENTS.md`. Both were prompt-layer mitigations. [Ca11back's reproduction on `#241`](https://github.com/mindfold-ai/Trellis/issues/241) showed the prompt-layer fix didn't reach reality:

- The main agent still spawned `trellis-research` with default `fork_turns="all"` despite the AGENTS.md rule.
- The child inherited the parent's transcript including prior `spawn_agent(...)` tool-call records.
- The child read AGENTS.md's "ALWAYS wait for every spawned subagent..." rule, applied it to _itself_, and called `wait_agent` on the inherited records.
- No agents to wait for → `No agents completed yet` → stuck.

Structural fix: each `packages/cli/src/templates/codex/agents/trellis-{implement,check,research}.toml` now contains:

```toml
[features]
multi_agent = false

[features.multi_agent_v2]
enabled = false
```

Codex's tool registration ([`codex-rs/tools/src/tool_config.rs:179-180`](https://github.com/openai/codex/blob/main/codex-rs/tools/src/tool_config.rs#L179-L180)) gates `spawn_agent` / `wait_agent` / `list_agents` / `close_agent` on `Collab || MultiAgentV2`. With both off, those tools don't exist in the sub-agent's tool list. The model literally cannot call `wait_agent` — there's no API to wishfully ignore.

This works because Codex agent role files support full `ConfigToml` override layers ([`codex-rs/core/src/config/agent_roles.rs:217-225`](https://github.com/openai/codex/blob/main/codex-rs/core/src/config/agent_roles.rs#L217-L225) — `RawAgentRoleFileToml` flattens `ConfigToml`), so `[features]` table entries override the base config when the role is applied at spawn time.

Adds `[issue-241-followup]` regression test in `packages/cli/test/regression.test.ts` asserting all three template toml files retain the disable block.

### Codex agent template drift — "Load Trellis Context First" port

The Codex sub-agent role files in `.codex/agents/` (project dogfood) had a 27-line "## Required: Load Trellis Context First" section telling the sub-agent how to find its active task path on class-2 platforms. The canonical template files in `packages/cli/src/templates/codex/agents/trellis-{implement,check}.toml` were missing it. Drift went undetected: dogfood was richer than what `trellis init` shipped to user projects.

Symptom users would see: Codex sub-agents on user projects don't know to parse `Active task: <path>` from the dispatch prompt, fall through to `task.py current --source`, and on session-id misalignment fall through again to "ask the user" — so they ask the user every time instead of auto-loading task context.

Fix: copy the section from `.codex/agents/trellis-implement.toml` and `.codex/agents/trellis-check.toml` into the template files of the same name. `trellis-research.toml` was already in sync. No behavior change for dogfood; user projects upgrading via `trellis update` will get the new section.

### Windows Python detection fallback chain ([`#236`](https://github.com/mindfold-ai/Trellis/issues/236))

`trellis init` previously tried only `python --version` on Windows. If the host had Python under `python3` (Microsoft Store) or `py -3` (python.org launcher) but not `python`, init failed outright with `Python command "python" not found`. The workaround was to install 0.4.0 first (less strict probe) then upgrade.

`resolveSupportedPython()` in `packages/cli/src/commands/init.ts` now walks a per-platform candidate list:

| Platform | Candidate order            |
| -------- | -------------------------- |
| Windows  | `python → python3 → py -3` |
| Other    | `python3 → python`         |

First candidate whose `--version` matches Python ≥ 3.9 wins. The resolved command is cached via `setResolvedPythonCommand()` in `configurators/shared.ts` so `replacePythonCommandLiterals()` and all downstream template / configurator writes pick up the same value.

Two env-var escape hatches:

- `TRELLIS_PYTHON_CMD=<cmd>` — explicit override, no probe.
- `TRELLIS_SKIP_PYTHON_CHECK=1` — skip probe entirely (pre-existing).

Failure case throws an aggregated error listing every candidate's probe result plus a Windows-specific install hint pointing at python.org with the "Add Python to PATH" reminder.

6 new unit tests in `packages/cli/test/commands/init-internals.test.ts` cover the fallback chain, env-var overrides, sandbox-restricted EPERM, and aggregated failure mode.

## Upgrade

```bash
npm install -g @mindfoldhq/trellis@latest
trellis update
```

No `--migrate` needed. Codex users get hardened sub-agent role files; Windows users no longer need to install 0.4.0 first.
