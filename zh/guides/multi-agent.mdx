## 为什么要多个 Agent

有些任务是独立的。加登录页面不影响加设置页面。每个跑一个 agent，就能并行。

Trellis 通过 Git worktree 支持这个。每个 agent 在你仓库的隔离副本里工作。开发过程中不会有合并冲突。

## 怎么工作的

```
主仓库
    │
    ├── worktree-1/  →  Agent 1 (feature-a)
    ├── worktree-2/  →  Agent 2 (feature-b)
    └── worktree-3/  →  Agent 3 (feature-c)
```

每个 worktree 是独立的 checkout，在独立的分支上。Agent 之间互不干扰。

完成后，每个 agent 创建一个 PR。你审查然后合并。

## 启动并行 Agent

在 Claude Code 里用 `/trellis:parallel` 命令。它会引导你：

- 定义任务
- 创建 worktree
- 启动 agent 会话

这会创建：

- 每个任务一个 worktree
- 每个任务一个 Claude Code 会话
- 每个任务一个分支

每个会话独立运行。

## 监控进度

查看状态：

```bash
python .trellis/scripts/multi_agent/status.py
```

输出会显示每个任务的当前阶段和分支。

## Agent 流水线

每个 agent 经过这些阶段：

1. **Plan** - 理解需求，找出要改的文件
2. **Implement** - 写代码
3. **Check** - 按规范审查，自己修问题
4. **PR** - 创建 pull request

check 阶段有个 "Ralph Loop"——如果发现问题，它会回去修。这防止提交质量差的 PR。

## 什么时候用

适合：

- 独立的功能
- 并行修 bug
- 生成变体（同一问题的不同方案）

不适合：

- 互相依赖的功能
- 改同一个文件的工作
- 需要跟 AI 反复迭代时

## 清理

完成后清理 worktree：

```bash
python .trellis/scripts/multi_agent/cleanup.py
```

这会删除 worktree 但保留分支。合并后的分支正常删就行。

## 限制

多 Agent 只支持 Claude Code。Cursor 不支持自动上下文注入需要的 hooks。

你也需要足够的 API 额度。三个 agent 意味着三倍的 token 消耗。
