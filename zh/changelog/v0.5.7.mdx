三个 Codex / Windows 修复。子代理 collab 工具从 prompt-layer 警告升级为协议层硬禁用（[`#240`](https://github.com/mindfold-ai/Trellis/issues/240) follow-up、[`#241`](https://github.com/mindfold-ai/Trellis/issues/241)）。把原来只在本地 dogfood 里的"Load Trellis Context First"段 port 回 template。Windows `trellis init` 改成 Python 命令 fallback chain，不再因首个候选失败就退出（[`#236`](https://github.com/mindfold-ai/Trellis/issues/236)）。无新增 migration。

## Bug Fixes

### Codex 子代理 collab 工具 — 协议层硬禁用（[`#240`](https://github.com/mindfold-ai/Trellis/issues/240) follow-up、[`#241`](https://github.com/mindfold-ai/Trellis/issues/241)）

[v0.5.5](/zh/changelog/v0.5.5) 移除了 `SessionStart` 注入路径。[v0.5.6](/zh/changelog/v0.5.6) 在 `AGENTS.md` 加了 prompt-layer 的 `fork_turns="none"` 提示。两条都是 prompt 层修复。[Ca11back 在 `#241` 下的复现](https://github.com/mindfold-ai/Trellis/issues/241)显示 prompt 层修复没落地：

- 主代理仍用默认 `fork_turns="all"` 派发 `trellis-research`，无视 AGENTS.md 规则。
- 子代理继承父对话历史，包含父之前的 `spawn_agent(...)` 工具调用记录。
- 子代理读到 AGENTS.md 的 "ALWAYS wait for every spawned subagent..." 规则，把它套到自己头上，对继承来的 spawn 记录调 `wait_agent`。
- 没有真实子代理可等 → `No agents completed yet` → 卡死。

结构性修复：每个 `packages/cli/src/templates/codex/agents/trellis-{implement,check,research}.toml` 现在带：

```toml
[features]
multi_agent = false

[features.multi_agent_v2]
enabled = false
```

Codex 工具注册（[`codex-rs/tools/src/tool_config.rs:179-180`](https://github.com/openai/codex/blob/main/codex-rs/tools/src/tool_config.rs#L179-L180)）以 `Collab || MultiAgentV2` 为门：两个都关 → `spawn_agent` / `wait_agent` / `list_agents` / `close_agent` 不会注册到子代理工具列表。模型**根本看不到这些工具**，无所谓"忽略 prompt"——根本没有 API 可调。

之所以可行：Codex agent role 文件支持完整 `ConfigToml` 覆盖层（[`codex-rs/core/src/config/agent_roles.rs:217-225`](https://github.com/openai/codex/blob/main/codex-rs/core/src/config/agent_roles.rs#L217-L225) — `RawAgentRoleFileToml` flatten 了 `ConfigToml`），`[features]` table 在 spawn 时会覆盖 base config。

新增 `packages/cli/test/regression.test.ts` 中 `[issue-241-followup]` 回归测试，断言三个 template toml 都保留 disable block。

### Codex agent 模板 drift — "Load Trellis Context First" 段补回

`.codex/agents/`（项目本地 dogfood）下的 Codex 子代理 role 文件原本就带 27 行 "## Required: Load Trellis Context First" 段，告诉子代理在 class-2 平台上怎么找到 active task 路径。但 `packages/cli/src/templates/codex/agents/trellis-{implement,check}.toml` 这两个 template 文件**缺这段**。drift 没被发现：dogfood 比 `trellis init` 实际派发给用户的内容更丰富。

用户能看到的症状：用户项目里的 Codex 子代理不知道去 dispatch prompt 第一行读 `Active task: <path>`，落到 `task.py current --source`，再因为 session-id 不匹配而落到"问用户"——所以**每次都问用户**，没法自动加载 task context。

修法：把 `.codex/agents/trellis-implement.toml` 和 `trellis-check.toml` 里的这段 copy 回同名 template 文件。`trellis-research.toml` 原本就一致。dogfood 不变；用户项目通过 `trellis update` 升级时会拿到新段。

### Windows Python 检测 fallback chain（[`#236`](https://github.com/mindfold-ai/Trellis/issues/236)）

`trellis init` 之前在 Windows 上只 try `python --version`。如果 host 上 Python 装在 `python3`（Microsoft Store）或 `py -3`（python.org launcher）下而没有 `python`，init 会直接失败 `Python command "python" not found`。临时 workaround 是先装 0.4.0（检查更宽松）再升 0.5.0+。

`packages/cli/src/commands/init.ts` 里的 `resolveSupportedPython()` 改成按平台 candidate list 顺序探测：

| 平台    | 候选顺序                   |
| ------- | -------------------------- |
| Windows | `python → python3 → py -3` |
| 其他    | `python3 → python`         |

第一个 `--version` 拿到 Python ≥ 3.9 的胜出。结果通过 `configurators/shared.ts` 的 `setResolvedPythonCommand()` 缓存，使 `replacePythonCommandLiterals()` 和所有后续 template / configurator 写入都用相同值。

两个 env-var 逃生口：

- `TRELLIS_PYTHON_CMD=<cmd>` — 显式覆盖，不探测。
- `TRELLIS_SKIP_PYTHON_CHECK=1` — 完全跳过探测（之前就有）。

全部失败时抛出聚合错误：列出每个候选的探测结果 + Windows 专属安装提示（指向 python.org，提醒勾选 "Add Python to PATH"）。

`packages/cli/test/commands/init-internals.test.ts` 加 6 个单测覆盖 fallback chain、env-var 覆盖、sandbox-restricted EPERM、聚合失败。

## Upgrade

```bash
npm install -g @mindfoldhq/trellis@latest
trellis update
```

不需要 `--migrate`。Codex 用户拿到加固后的 sub-agent role 文件；Windows 用户不再需要先装 0.4.0。
