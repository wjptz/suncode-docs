## 多平台与团队配置

Suncode 支持 21 个平台（Claude Code、Cursor、OpenCode、Engineer、Codex、Kiro、Kilo、Gemini CLI、Antigravity、Devin（原 Windsurf）、Qoder、CodeBuddy、GitHub Copilot、Droid、Pi Agent、Oh My Pi、Reasonix、ZCode、Trae、Grok Build 和 Kimi Code），以及任何读取 `.agents/skills/` 规范的 AI 编码 agent（Amp、Cline、Deep Agents、Firebender、Warp 等）。`.suncode/` 核心跨平台一致；差异在于**各平台如何交付 hook、extension、skill、sub-agent、command**。

### 加入已初始化的 Suncode 项目

项目已经被别人 `suncode init` 过了，你作为新成员加入——直接跑 `suncode init`，CLI 会识别出已初始化，给你三个选项：

```
? Suncode is already initialized. What would you like to do?
❯ Add AI platform(s)
  Set up developer identity on this device
  Full re-initialize
```

| 选项                                         | 何时选                                                             |
| -------------------------------------------- | ------------------------------------------------------------------ |
| **Add AI platform(s)**                       | 想给当前项目加一个别人没装的平台（如你用 Cursor、但别人只装了 CC） |
| **Set up developer identity on this device** | 你是新成员，只要把**自己的身份**写到本机                           |
| **Full re-initialize**                       | 项目配置坏了，重来一遍                                             |

新成员选第 2 个。CLI 会问你想用的开发者名（默认取自 Git user），然后：

- 写 `.suncode/.developer` 记录你的开发者名（gitignored，每台机器独立）
- 在 `.suncode/workspace/<your-name>/` 下建你自己的 journal 目录

完成后，在支持 SessionStart hook 或 extension 的平台上开新会话就自动注入 Suncode 上下文；没有自动会话注入的平台跑 `/suncode:start` 或对应平台的 start workflow。

<Note>
  不要选 **Full re-initialize**——那会覆盖 `.suncode/`、`.claude/` 等已有配置，影响全团队。
</Note>

图例：✅ Suncode 写入配置并由平台执行 · ⚡ 部分支持（Codex 使用 `AGENTS.md` 加
`UserPromptSubmit`；workflow breadcrumb 需要 `features.hooks = true`，0.129+
还要走一次 `/hooks` 审批；Kiro 在 `suncode` agent 上带了每回合 `userPromptSubmit` hook，但用户需激活该 agent —— `chat.defaultAgent suncode` 或 `/agent swap`） · ❌ 平台不暴露这个事件 ·
— 该平台没有命令原语，start / finish-work / continue 作为 skill 交付。Qoder 用的是
`/suncode-{name}`（连字符，不是冒号），通过原生 Custom Commands 交付 `finish-work` /
`continue`；`start` 由 SessionStart hook 自动注入。Engineer 使用和 OpenCode 一样的
OpenCode-compatible JS plugin surface，只是放在 `.engineer/` 下（OpenCode 放在 `.opencode/`）。Kimi 通过 `/skill:suncode-*` 命令 skill 调用。Pi Agent 和 Oh My Pi 走 extension，不走 Python hook 文件；会话上下文、Bash 环境变量传递和 sub-agent context 都在 agent 动作前完成。

`.kilocode/`、`.agent/`（Antigravity）、`.devin/` 只有 workflow 和 skill，没有 sub-agent 也没有 hook。`.agents/skills/` 是 Codex、Gemini CLI、Pi Agent、Kimi Code 和 ZCode 写入的共享 skill 层。

### Claude Code

自动化最完整。Hook 布局：

| Hook                         | 触发               | 效果                                        |
| ---------------------------- | ------------------ | ------------------------------------------- |
| `session-start.py`           | SessionStart       | 注入身份、git 状态、活跃任务                |
| `inject-workflow-state.py`   | UserPromptSubmit   | 提示 AI 遵守当前任务的状态契约              |
| `inject-subagent-context.py` | PreToolUse（Task） | 加载 `implement.jsonl` / `check.jsonl` / 等 |

Sub-agent：`suncode-implement`、`suncode-check`、`suncode-research`，在 `.claude/agents/`。
Skill：`suncode-brainstorm`、`suncode-before-dev`、`suncode-check`、`suncode-update-spec`、`suncode-break-loop`，在 `.claude/skills/`。
Command：`start`、`finish-work`、`continue`，在 `.claude/commands/suncode/`。

### Cursor

```bash
suncode init -u your-name --cursor
```

Cursor 是完整 class-1 平台：真 hook、真 sub-agent、真 skill。

布局：

- `.cursor/commands/suncode-{name}.md`：`start`、`finish-work`、`continue`（扁平命名加 `suncode-` 前缀，调用形式 `/suncode-start` 等）
- `.cursor/skills/suncode-{name}/SKILL.md`：5 个 suncode skill
- `.cursor/agents/`：`suncode-implement.md`、`suncode-check.md`、`suncode-research.md`
- `.cursor/hooks/`：共享 Python hook 脚本（与 Claude Code 兼容）
- `.cursor/hooks.json`：hook 配置（Cursor 用独立文件，不是 `settings.json`）

### OpenCode

```bash
suncode init -u your-name --opencode
```

OpenCode 1.2.x 是 class-1 平台（真 hook + 真 sub-agent）：

- `.opencode/commands/suncode/`：start / finish-work / continue
- `.opencode/agents/`：`suncode-implement.md`、`suncode-check.md`、`suncode-research.md`
- `.opencode/skills/`：5 个 suncode skill
- `.opencode/plugins/`：JS 插件：`session-start.js`、`inject-subagent-context.js`、`inject-workflow-state.js`

插件是 factory function（OpenCode 1.2+）。Suncode 生成的版本对齐 Claude Code 行为。

### Engineer

```bash
suncode init -u your-name --engineer
```

Engineer 兼容 OpenCode。Suncode 会把同一套 command、sub-agent、skill 和 JS plugin 写到 `.engineer/`，因此同一仓库可以同时保留独立的 `.opencode/` 配置。

- `.engineer/commands/suncode/`：start / finish-work / continue
- `.engineer/agents/`：`suncode-implement.md`、`suncode-check.md`、`suncode-research.md`
- `.engineer/skills/`：5 个 suncode skill
- `.engineer/plugins/`：JS 插件：`session-start.js`、`inject-subagent-context.js`、`inject-workflow-state.js`

### Codex

```bash
suncode init -u your-name --codex
```

布局：

- `AGENTS.md`（仓库根）：入口文件，Codex 每个 session 自动读取（作为 prelude）
- `.agents/skills/`：中立渲染的共享 Suncode 命令和工作流 skill
- `.codex/skills/`：Codex 专用 skill override
- `.codex/agents/`：TOML sub-agent：`suncode-implement.toml`、`suncode-check.toml`、`suncode-research.toml`
- `.codex/hooks/` + `.codex/hooks.json`：`UserPromptSubmit` workflow-state 与三个 Suncode 角色的原生 `SubagentStart` 上下文
- `.codex/config.toml`：包含 `[agents].max_depth = 1` 的项目默认配置
- `.codex/hooks/session-start.py`：保留的紧凑 SessionStart 兼容脚本，默认不 wiring

<Note>
Codex hook 需要开启。在 `~/.codex/config.toml` 加：

```toml
[features]
hooks = true   # Codex 0.129+。旧版用 `codex_hooks = true`。
```

Codex 0.129+ 还要在 TUI 里跑一次 `/hooks` 并审批 Suncode hooks。Suncode 把 `codex.dispatch_mode: inline` 作为默认及缺失/非法值回退；显式设为 `auto`（旧别名 `sub-agent`）时使用原生 sub-agent dispatch。`SubagentStart` 严格按父 session 注入上下文，agent profile 仍保留 hook 不可用时的 pull fallback。

Codex channel worker 可用 `suncode channel spawn <channel> --provider codex --sandbox <mode>` 选择 `read-only`、`workspace-write`（默认）或 `danger-full-access`；非 Codex provider 忽略此选项。

</Note>

### Kiro

```bash
suncode init -u your-name --kiro
```

布局：

- `.kiro/agents/suncode.json`：主 Suncode agent —— 每回合 `userPromptSubmit` hook + 开场 `agentSpawn` hook + 把 `.suncode/workflow.md` 作为常驻 resource
- `.kiro/agents/suncode-{implement,check,research}.json`：子代理（`agentSpawn` 注入任务上下文）
- `.kiro/hooks/*.py` + `.kiro/hooks/suncode-workflow-state.kiro.hook`：每回合 workflow-state 注入器（CLI agent hook + IDE `.kiro.hook`）
- `.kiro/skills/*/SKILL.md`：auto-trigger skill

**启用（必做，否则工作流不会激活）：**

- **Kiro CLI**：把 `suncode` 设为当前 agent，它的 hook 才会触发 —— `kiro-cli settings chat.defaultAgent suncode`（持久）或 `/agent swap suncode`（按 session）。否则 Kiro 跑内置 `kiro_default`。
- **Kiro IDE**：`.kiro/hooks/suncode-workflow-state.kiro.hook`（一个 `promptSubmit` hook）默认 enabled；在 Kiro 的 Agent Hooks 界面确认它已开启/信任。

<Note>
  每回合注入打印纯文本，Kiro 会把它加入对话上下文（依 Kiro 官方 hooks
  文档）。stdout→上下文的确切行为（以及 IDE `runCommand` 是否注入 stdout）待 Kiro
  真机验证；若不生效，退化为静态 steering 提示。
</Note>

### Gemini CLI

```bash
suncode init -u your-name --gemini
```

布局：

- `.gemini/commands/suncode/{name}.toml`：TOML 命令文件——`start.toml`、`finish-work.toml`、`continue.toml`
- `.agents/skills/suncode-{name}/SKILL.md`：Gemini CLI 0.40+ 作为 workspace alias 读取的共享 Suncode skill
- `.gemini/agents/{name}.md`：带 pull-based prelude 的 sub-agent 定义（sub-agent 自己 `Read` 自己的 JSONL，因为 Gemini 没有 sub-agent `PreToolUse` hook）
- `.gemini/hooks/session-start.py`：SessionStart hook

### Qoder

```bash
suncode init -u your-name --qoder
```

Qoder 原生支持 Custom Commands（在 Agent 输入框输入 `/` 调用，项目级存放在 `<项目>/.qoder/commands/{name}.md`，要求 YAML frontmatter 包含 `name` 和 `description` 字段）。Suncode 把 session-boundary 命令放到这里，让你可以显式地调用；阶段级工作流则保留为 auto-trigger skill。

布局：

- `.qoder/commands/suncode-{name}.md`：session-boundary 命令 —— `finish-work`、`continue`，用 `/suncode-finish-work`、`/suncode-continue` 调用（`start` 由 Qoder 的 SessionStart hook 自动注入，无需命令）
- `.qoder/skills/suncode-{name}/SKILL.md`：auto-trigger 工作流 skill —— `brainstorm`、`before-dev`、`check`、`update-spec`、`break-loop`
- `.qoder/agents/{name}.md`：带 pull-based prelude 的 sub-agent 定义
- `.qoder/hooks/session-start.py`：SessionStart hook
- `.qoder/settings.json`：hook 配置（只有 SessionStart——Qoder 没有 sub-agent `PreToolUse` hook）

### GitHub Copilot

- `.factory/commands/suncode/`：start / finish-work / continue

- `.factory/skills/`：5 个 suncode skill

### Pi Agent

```bash
suncode init -u your-name --pi
```

Pi Agent 走 extension，不走 Python hook。Suncode 写入同样的 workflow 原语，然后 extension 解析当前 session id，在 Bash 命令和 sub-agent 运行前注入任务上下文。

布局：

- `.pi/prompts/suncode-{name}.md`：会话边界 prompt（`finish-work`、`continue`；`start` 只在需要手动入口的平台存在）
- `.agents/skills/suncode-{name}/SKILL.md`：5 个中立渲染的共享 Suncode 工作流 skill
- `.pi/agents/{name}.md`：`suncode-implement`、`suncode-check`、`suncode-research`
- `.pi/extensions/suncode/index.ts`：会话上下文、Bash `SUNCODE_CONTEXT_ID` 传递、sub-agent JSONL 注入
- `.pi/settings.json`：extension 注册

extension 把当前任务状态存在 `.suncode/.runtime/sessions/<session-key>.json`，所以每个 Pi 窗口/会话都可以有自己的任务，不会抢别的窗口。

### Oh My Pi、Grok Build 与 Kimi Code

```bash
suncode init -u your-name --omp --grok --kimi
```

- **Oh My Pi** 写入带 Suncode 命名空间的 command、skill、agent 和
  `.omp/extensions/suncode/index.ts`。共享 `.omp/` 根只有在存在 Suncode
  所有权 marker 时才算已配置。Bash bridge 只向该次调用的 env 注入
  `SUNCODE_CONTEXT_ID`，保留用户显式值且不改 command。
- **Grok Build** 不安装 hook，走 pull-based：写入扁平
  `.grok/commands/suncode-*.md`、`.grok/skills/` 和 `.grok/agents/`，通过
  `spawn_subagent` 指定 Suncode 角色。
- **Kimi Code** 把共享工作流 skill 写入 `.agents/skills/`，把私有命令/角色
  skill 写入 `.kimi-code/skills/`。它不安装项目级 hook/settings，通过 Kimi
  内置 `coder` / `explore` agent 配合对应 Suncode 角色指令执行。

### 其他支持平台

Suncode 还原生带有上面能力矩阵之外几个平台的配置器：

- **Kilo**（`--kilo`）：写 `.kilocode/workflows/`（命令：`start`、`finish-work`）和 `.kilocode/skills/suncode-{name}/SKILL.md`（5 个 suncode skill）。没有 hook 集成。
- **Antigravity**（`--antigravity`）：写 Antigravity 原生的 workflow 文件（3 个命令）。
- **Devin**（`--devin`，原 Windsurf）：写 Devin 原生的 workflow 和 skill 文件。旧的 `--windsurf` flag 仍作为已废弃的别名可用。
- **Reasonix**（`--reasonix`）：写 `.reasonix/skills/`，其中 workflow 作为 skill，implement/check 用带 `runAs: subagent` frontmatter 的 sub-agent skill。没有 hook 集成。
- **ZCode**（`--zcode`）：写 `.zcode/commands/suncode/`、`.zcode/cli/agents/` 和 `.agents/skills/`。Sub-agent 走 pull-based prelude。
- **Trae**（`--trae`）：写 `.trae/commands/`、`.trae/skills/`、`.trae/agents/`、`.trae/hooks/` 和 `.trae/hooks.json`。主会话上下文走 SessionStart/UserPromptSubmit hook，sub-agent 走 pull-based prelude。

除了 21 个已配置平台，Suncode 还能被任何遵循 `.agents/skills/` 约定（[agentskills.io](https://agentskills.io) 规范）的 AI 编码 agent 直接使用。Codex、Gemini CLI、Pi Agent、Kimi Code 和 ZCode 会向该目录写入字节兼容的 skill，Amp、Cline、Deep Agents、Firebender、Warp 等生态 agent 可直接读取。

### 多开发者协作

按开发者隔离（不冲突）：

- `.suncode/workspace/{name}/`：各自的 journal 和索引
- `.suncode/.developer`：gitignored
- `.suncode/.runtime/`：gitignored 的会话运行时状态；每个 AI 会话/窗口有自己的 active task 文件

共享状态（通过 PR 协调）：

- `.suncode/spec/`：团队约定，和代码一样走 PR review
- `.suncode/tasks/`：task JSON；显式 `--assignee` 避免冲突

Spec 的重要改动应在 review 中讨论；把 spec 库当成团队代码来对待。

### `suncode update` 与版本管理

```bash
cat .suncode/.version          # 当前版本
suncode update                  # 升级到最新
suncode update --dry-run        # 预演
suncode update --migrate        # 应用破坏性变更的 migration（大版本必需）
suncode update -f               # 强制覆盖本地改过的文件
suncode update -s               # 跳过本地改过的文件
```

模板 hash 机制（`.suncode/.template-hashes.json`）：

1. 计算本地文件 hash。
2. 和记录的模板 hash 对比。
3. 一致 ⇒ 用户未改 ⇒ 安全更新。
4. 不一致 ⇒ 询问（覆盖 / 跳过），或根据 `-f` / `-s` 策略静默处理。

破坏性变更（比如 0.5.0 移除 Multi-Agent Pipeline）以 migration manifest 形式发布。对破坏性 manifest 跑 `suncode update` 而不加 `--migrate` 会退出并打印说明，不会静默重命名。`suncode update --migrate` 应用 rename / delete 条目，每个冲突询问一次。
