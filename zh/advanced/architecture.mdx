Trellis 的产品定位是 **Team-level Agent Harness with built-in LLM wiki** —— 在团队仓库里的 AI 工程化外壳 + LLM 直接读的团队知识库。本页给开发者讲整套系统怎么运转、每个模块怎么扩。普通用户使用流程看 [How it works](/start/how-it-works)。

## 设计理念

Trellis 把 AI coding 当工程问题，不当聊天问题。

- **不靠记忆** — spec 由 hook / sub-agent prelude 主动注入，AI 不需要"记得"读
- **不靠对话** — 知识落到文件（`spec/` / `tasks/` / `workspace/`），对话压缩了也不会丢
- **不靠超人** — 多 sub-agent 分工，每个 agent 范围有限职责清晰
- **不靠信任** — AI 不提交代码，commit 边界由人把守
- **不靠通用** — 团队规范是"项目语料"，不是 Stack Overflow 答案

## Feature 全景

围绕 vision 拆三条主线：harness 治 AI 行为、wiki 治团队知识、team-level 是这两件事的协作放大器。

### Agent Harness（约束 AI 行为的工程化外壳）

- **14+ AI coding 平台原生适配** — Claude Code / Cursor / Codex / Gemini / Qoder / OpenCode / Copilot / CodeBuddy / Droid / Pi / Kiro / Kilo / Antigravity / Windsurf。每个平台用各自的 hook / skill / extension / workflow 原语，同一份 `.trellis/` 行为一致
- **三 Phase 工作流状态机** — Plan / Execute / Finish，每步都有 breadcrumb 在主会话每轮注入；AI 跳步被 hook 拽回来
- **Sub-agent 三层执行** — `trellis-research`（调研）/ `trellis-implement`（写代码）/ `trellis-check`（验收 + 自修），分工 + 互相约束
- **强制 read-before-write** — `before-dev` skill / `implement.jsonl` 注入 / pull-based prelude 三种机制，sub-agent 写代码前先读本任务相关的 spec
- **AI 不提交代码** — harness 边界：sub-agent 写完，人类验收 commit；每次变更都过 PR review

### Built-in LLM Wiki（团队知识沉淀，AI 自动读）

- **`.trellis/spec/`** — 团队规范库；按 package / layer 组织，sub-agent 通过任务 `implement.jsonl` 按需注入，主会话不需要 dump 全量
- **`.trellis/tasks/`** — 任务级知识；每个任务自带 `prd.md` + `research/*.md` + `info.md` + jsonl context manifests
- **`.trellis/workspace/`** — 开发者级 journal；跨 session 复盘、记录决策、避免重复踩坑
- **`.trellis/workflow.md`** — 工作流自身的 wiki；单一事实源，改一个文件整个项目的工作流就改了

### Team-level（协作维度）

- **`.trellis/` 是团队的，不是个人的** — workflow / spec / tasks 全部 git-tracked，新开发者 `git clone` 拿到完整 harness + 团队知识库；首次跑 `trellis init` 自动建 joiner task 引导上手
- **多任务并行 + 子任务树** — 同一仓库同时跑多个任务，subtask 分支汇总到 parent；active task pointer 按 session 隔离，多窗口互不串
- **跨开发者知识沉淀** — 一个人在任务里写的 PRD + research、归档到 `tasks/archive/` 的产物、提炼到 `spec/` 的规范，全团队下次的 sub-agent 都能自动读到（不只是你自己）
- **每开发者独立 workspace** — `.trellis/workspace/<dev>/` 按 developer 隔离 journal 不互相覆盖；`spec/` 是团队共享 SoT

## `workflow.md`：单一事实源

`.trellis/workflow.md` 是 Plan / Execute / Finish 契约的唯一定义文件。三个 Phase 的所有 step（如 1.0 `task.py create` → 1.4 `task.py start` → 2.1 `trellis-implement` → 2.2 `trellis-check` → 3.4 `/trellis:finish-work`）都写在这里。系统其他部件都从它读：

- **SessionStart hook** 自动注入它的 TOC + step body，开会话前 AI 已经拿到契约
- **`/trellis:continue`** 读当前会话 runtime（`.trellis/.runtime/sessions/<session-key>.json`）+ 任务 `status`，对照 `workflow.md` 判断当前 step 和下一 step
- **每轮 `<workflow-state>` 面包屑**（`inject-workflow-state.py` hook）每条用户消息触发一次，按当前 task `status` 找到 `[workflow-state:STATUS]` 块并把正文注入。Hook 只解析、不嵌内容；找不到对应块时输出固定一行 `Refer to workflow.md for current step.`
- **Sub-agent 注入** `workflow.md` 是 `implement.jsonl` 默认第一条，sub-agent 也能拿到约定

**任务状态默认 3 种**：`planning` / `in_progress` / `completed`，分别由 `task.py create` / `start` / `archive` 写入，跟三个 Phase 一一对应。`task.py create` 同时把当前 session 的 active-task pointer 指向新任务，`[workflow-state:planning]` 从下一轮就开始生效。`workflow.md` 里按状态分块写提醒；要加自定义状态（如 `review` / `blocked`），直接加对应 `[workflow-state:STATUS]...[/workflow-state:STATUS]` 块——不用改代码。

> 改 `workflow.md` 就改了整个项目的工作流，下次 session 所有平台、skill、sub-agent、命令全按新版读。别处没有工作流硬编码。

**接入分层**：

- **Hook + sub-agent**（最完整）：Claude Code、Cursor、OpenCode、Gemini、Qoder、CodeBuddy、Copilot、Droid、Pi、iFlow —— 自动注入 + sub-agent 隔离 + 每轮面包屑

## 会话启动流程

<Tabs>
  <Tab title="Hook / extension 可用（CC / Cursor / OC / Gemini / Qoder / CodeBuddy / Copilot / Droid / Pi）">
    SessionStart hook 或 extension 在 session 启动时自动触发，平台对应的 `session-start.py` /
    `session-start.js` / Pi extension 不需要手动调用即完成 Trellis
    上下文注入。打开终端就直接描述任务；暴露 `/trellis:start` 的平台可以用它显式查看上下文报告。
  </Tab>
  <Tab title="Codex">
    Codex 每个 session 自动读仓库根的 `AGENTS.md`。在 `~/.codex/config.toml` 里打开 `codex_hooks =

  </Tab>
  <Tab title="Kiro">
    Kiro 通过 `.kiro/` 下的 skill 文件交付 Trellis 内容。Kiro 的 Agent Hook

    `@trellis:start`。
  </Tab>
  <Tab title="Kilo / Antigravity / Windsurf">
    这三个平台只有 workflow + skill，没有 hook。手动跑入口 workflow（`/start.md` /
    `.agent/workflows/start.md` / `/trellis-start`），AI 内联读 `workflow.md` + `get_context.py` +
    spec 索引后询问你要做什么。
  </Tab>


## 规范注入机制

<Info>
  Sub-agent 自动规范注入依赖平台的 PreToolUse hook 或 extension 等价机制（"hook-push"）。Codex /
  Copilot / Gemini / Qoder 走 prelude 模式（"pull-based"），sub-agent 在第一轮自己 Read
  上下文。两种路径都在共享基础设施里。
</Info>

### Hook-push 流程（Claude Code / Cursor / OpenCode / Kiro / CodeBuddy / Droid）

```
Main session spawns: Task(subagent_type="trellis-implement", prompt="...")
     │
     ▼
┌───────────────────────────────────────────────────────────────┐
│  Hook / extension: inject-subagent-context                    │
│  (在 sub-agent 启动前运行)                                     │
│                                                               │
│  读取发生在 hook 里，不在 sub-agent 里：                       │
│                                                               │
│  1. 读取当前会话 runtime                                      │
│     → .trellis/.runtime/sessions/<session-key>.json           │
│     → current task directory path                             │
│                                                               │
│  2. 读取 {task_dir}/implement.jsonl                           │
│     → list of spec files this sub-agent needs                 │
│                                                               │
│  3. 读取 JSONL 引用的每个文件                                 │
│     → .trellis/spec/backend/index.md                          │
│     → .trellis/spec/backend/database-guidelines.md            │
│                                                               │
│  4. 读取 prd.md（需求）和 info.md（设计，可选）                │
│                                                               │
│  5. 拼接全部内容作为 additionalContext 给 sub-agent           │
└───────────────────────────────┬───────────────────────────────┘
                                │
                                ▼
       Sub-agent 启动时 prompt 已带这些内容，自己不读任何文件
```

### Pull-based prelude 流程（Codex / Copilot / Gemini / Qoder）

这些平台无法可靠地通过 hook 改写 sub-agent prompt（上游限制）。改走 sub-agent 定义最前面的 prelude：sub-agent 在第一轮自己 Read `.trellis/.runtime/sessions/<session-key>.json` → `prd.md` → `implement.jsonl`，再 Read 每个 jsonl 引用的文件。结果跟 hook-push 等价，只是发生时机往后挪了一轮。

### JSONL 配置格式

JSONL（JSON Lines）定义 hook 要替 sub-agent 注入哪些文件。每行一个 JSON 对象：

```jsonl
{"file": ".trellis/workflow.md", "reason": "项目工作流和约定"}
{"file": ".trellis/spec/backend/index.md", "reason": "后端开发指南"}
{"file": "src/services/auth.ts", "reason": "现有 auth 实现，要对齐模式"}
{"file": ".trellis/tasks/02-27-user-login/research/", "type": "directory", "reason": "调研产物"}
```

| 字段     | 必填 | 说明                                                                                                                                                    |
| -------- | ---- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `file`   | 是   | 文件或目录的相对路径（相对项目根）                                                                                                                      |
| `reason` | 是   | 为什么要读这个文件                                                                                                                                      |
| `type`   | 否   | 默认 `"file"`。单文件模式按原样读引用文件，不限扩展名（源码、JSON 都行）。`"directory"` 模式按文件名排序注入目录里的 `.md`（最多 20 个），非 `.md` 跳过 |

**三种 JSONL 文件**：

| 文件              | 使用者                     | 典型内容                               |
| ----------------- | -------------------------- | -------------------------------------- |
| `implement.jsonl` | `trellis-implement`        | workflow.md + 相关 spec + 代码模式范例 |
| `check.jsonl`     | `trellis-check`            | 质量 checklist spec + 测试 / lint 规则 |
| `research.jsonl`  | `trellis-research`（可选） | 项目结构概览 + 调研目标                |

JSONL 里放什么，取决于 sub-agent 执行这个任务需要什么：

- **Spec 和 workflow**：`.trellis/workflow.md`、`.trellis/spec/**/index.md`、具体指南文件 —— 塑形 sub-agent **怎么** 写代码
- **现有代码作为参照**：指向单个源码文件（`src/services/auth.ts`）让 sub-agent 对齐现有实现模式。优先单文件
- **调研产物**：目录模式指向 `{task_dir}/research/`，把 `trellis-research` 产出的每个 `.md` 全拉进来

## Sub-agent 系统

Trellis 0.5 原生 3 个 sub-agent。每个角色明确、约束清晰、有独立的 JSONL spec 注入通道。

| Sub-agent           | 角色                      | 约束                           | 主要上下文                   |
| ------------------- | ------------------------- | ------------------------------ | ---------------------------- |
| `trellis-research`  | 代码 / 文档搜索、技术调研 | 只读                           | `research.jsonl`             |
| `trellis-implement` | 写代码                    | 禁止 `git commit`              | `implement.jsonl` + `prd.md` |
| `trellis-check`     | 验证 + 自修复             | 只为修复才写代码；自带重试循环 | `check.jsonl` + `prd.md`     |

> 0.4 时期的 6 个 sub-agent（`dispatch` / `plan` / `debug` / `research` / `implement` / `check`）和 SubagentStop 上的 Ralph Loop 已在 0.5 移除：dispatch / plan / debug 由 skill routing 替代，Ralph Loop 由 `trellis-check` 内部重试循环替代。

**主会话中的典型任务流**：

```
trellis-brainstorm skill
  │  澄清范围、起草 prd.md、创建任务目录 + JSONL 配置
  │  需要调研时并行 spawn trellis-research sub-agent
  ▼
trellis-before-dev skill
  │  编码前先读相关 spec
  ▼
Spawn trellis-implement sub-agent
  │  Hook 注入 implement.jsonl 上下文
  │  Sub-agent 写代码，不 git commit
  ▼
Spawn trellis-check sub-agent
  │  Hook 注入 check.jsonl 上下文
  │  跑 lint / typecheck / test，对照 quality checklist 比对 diff
  │  内部重试循环里自修复
  ▼
[人工验收 / 真实环境业务测试]
  │
  ▼
[Phase 3.4：人工 commit]
  │
  ▼
/trellis:finish-work
  │  归档任务 + 把 session 条目追加到 workspace journal
```

## Workspace 系统

跨会话承载个人工作脉络，每个开发者隔离。

```
.trellis/workspace/
├── index.md                # 全局索引（所有 developer 列表）
└── {developer}/            # 按开发者隔离
    ├── index.md            # 个人 journal 索引（含 @@@auto markers 自动维护）
    └── journal-N.md        # 会话日志（约每 2000 行新起一个文件）
```

**关键文件**：

| 文件                                            | 作用                                     | gitignore         |
| ----------------------------------------------- | ---------------------------------------- | ----------------- |
| `.trellis/.developer`                           | 开发者身份（唯一标识）                   | ✅ 是（每人本地） |
| `.trellis/.current-task`                        | 主会话当前 active task 指针              | ✅ 是             |
| `.trellis/.runtime/sessions/<session-key>.json` | 会话级 task 指针（多窗口并行不互相污染） | ✅ 是             |
| `workspace/{name}/journal-*.md`                 | 会话日志正文                             | ❌ 否（团队共享） |
| `workspace/{name}/index.md`                     | 个人索引（自动维护）                     | ❌ 否             |

> Journal 切片规则：单文件超过 2000 行自动起新文件 `journal-N+1.md`，老 journal 保留可读。`/trellis:finish-work` 调用 `add_session.py` 写入新条目。

## Task 系统

任务是 AI 工作单元的事实源。每个 task 一个独立目录，承载需求、研究、上下文清单和元数据。

```
.trellis/tasks/{MM-DD-slug}/
├── task.json              # 元数据：status / phase / branch / assignee / 子任务
├── prd.md                 # 需求 + 验收标准 + non-goals
├── info.md                # 技术设计（可选）
├── implement.jsonl        # implement sub-agent 上下文清单
├── check.jsonl            # check sub-agent 上下文清单
├── research.jsonl         # research sub-agent 上下文清单（可选）
└── research/              # 调研产物（trellis-research 写入）
    ├── overview.md
    └── *.md
```

**`task.json` 关键字段**：

| 字段                   | 含义                                                 |
| ---------------------- | ---------------------------------------------------- |
| `id` / `slug`          | 任务标识                                             |
| `status`               | `planning` / `in_progress` / `completed`（自动维护） |
| `phase`                | 当前 step（如 `2.1`、`3.4`）                         |
| `branch`               | 关联 git 分支                                        |
| `creator` / `assignee` | 创建者 / 指派开发者                                  |
| `parent` / `children`  | 父子任务（subtask 树）                               |
| `prd_path` / `pr_url`  | 关联 PRD 和 PR                                       |

**Task lifecycle hooks**（`.trellis/config.yaml`）：

```yaml
hooks:
  after_create:
    - command: "linear-issue create --title '{task.title}'"
  after_start:
    - command: 'linear-issue update --status in_progress'
  after_finish:
    - command: 'linear-issue update --status done'
  after_archive:
    - command: 'linear-issue close'
```

四个事件触发点都能挂自定义 shell 命令，用来做 Linear / Jira / 内部系统同步。每个 hook 失败不阻塞主流程。

**`task.py` 16 个 subcommand**（按用途分组）：

| 分组      | 命令                                                       |
| --------- | ---------------------------------------------------------- |
| Lifecycle | `create` / `start` / `finish` / `archive` / `restore`      |
| Context   | `add-context` / `remove-context` / `list-context`          |
| Metadata  | `set-status` / `set-phase` / `set-assignee` / `set-branch` |
| Hierarchy | `add-subtask` / `remove-subtask`                           |
| PR        | `set-pr` / `unset-pr`                                      |

跑 `python3 ./.trellis/scripts/task.py --help` 看完整签名。

## 多任务并行

Trellis 把"多任务并行"当默认场景设计，不是事后补丁。三套机制叠在一起：

### 1. 任务级隔离

每个 task 独立目录 `.trellis/tasks/<MM-DD-slug>/`，PRD、`research/`、`implement.jsonl`、`check.jsonl` 各自写死。Sub-agent 只读当前 active task 的 jsonl，所以跨 task 的 spec 注入自动按任务范围分隔 —— task A 的失败结论不会污染 task B 的实现。

### 2. 会话级隔离（多窗口安全）

```
.trellis/.runtime/sessions/
├── 7f3a2b1c4d5e.json      # window 1 active task: 04-26-add-audit-log
├── 8e1d9c2a6f3b.json      # window 2 active task: 04-27-monorepo-migrate
└── ...
```

打开多个 IDE 窗口 / 终端会话同时跑 Trellis 时，每个会话有独立的 `<session-key>.json` 跟踪自己的 active task。SessionStart hook 在创建会话时分配一个 key（基于 PID + 时间戳），后续所有 hook 调用、`/trellis:continue`、sub-agent 注入都按这个 key 读自己的指针。**A 窗口切到 task X 不会让 B 窗口的 task Y 被打断**。

`.trellis/.current-task` 是兜底全局指针，用于 `task.py` 命令行用户 fallback。会话级指针优先。

### 3. 子任务树

复杂 task 可以拆子任务：

```
.trellis/tasks/
├── 04-26-payment-revamp/           # parent
│   ├── task.json                   # children: [04-26-payment-stripe, 04-26-payment-alipay]
│   └── prd.md
├── 04-26-payment-stripe/           # child
│   ├── task.json                   # parent: 04-26-payment-revamp
│   └── prd.md
└── 04-26-payment-alipay/           # child
    └── ...
```

Parent task 不直接写代码，承载需求和研究；child task 各自走完整 Phase 1-3 闭环。`task.py add-subtask <parent> <child>` 建立关系，brainstorm phase 在创建复杂任务时通常会自动拆分。

### 4. 跨任务知识流转

并行的 task 之间不共享上下文，但有三条流转通道（按持久度排序）：

| 通道                           | 作用                                             | 流转范围                                   |
| ------------------------------ | ------------------------------------------------ | ------------------------------------------ |
| `.trellis/spec/`               | 通过 `/trellis:update-spec` 沉淀稳定结论         | 所有未来 task 自动可见（jsonl 列了就注入） |
| `workspace/<dev>/journal-*.md` | 个人 session 脉络                                | 同一开发者的后续会话                       |
| Task PRD 显式引用              | task B 的 `prd.md` 链接 task A 的 PRD / research | 仅在 brainstorm 时被显式拉入               |

**不会自动流转的**：task A 的 `research/` 不会自动注入到 task B。结论稳定的就 `update-spec` 提到 spec；任务特有的留在源 task 里、新 task 显式引用。

### 关于 `/parallel` 命令

0.4 时期有 `/parallel` 命令 + `.trellis/scripts/multi_agent/`（Git worktree 编排，多个 agent 各自在 worktree 里跑）。0.5 移除了 —— 现在主流 agent CLI / IDE（Claude Code、Cursor、OpenCode 等）都有原生 worktree 支持，Trellis 不再自己造一套。想多 agent 并行写代码：用平台原生的 worktree 入口创建独立 worktree，每个 worktree 里跑独立的 Trellis 会话即可。

## Spec 系统

按团队规范注入 sub-agent 的事实源。支持单仓和 monorepo 两种布局。

```
# 单仓
.trellis/spec/
├── frontend/                  # 前端规范
│   ├── index.md               # ← 单行目录，列出本目录所有文件
│   ├── components.md
│   └── state-management.md
├── backend/                   # 后端规范
│   ├── index.md
│   ├── error-handling.md
│   └── database.md
└── guides/                    # 思考型 guide（跨前后端）
    ├── index.md
    └── cross-layer-thinking-guide.md

# Monorepo（按包隔离）
.trellis/spec/
├── <package-name>/
│   ├── backend/
│   ├── frontend/
│   └── unit-test/
└── guides/                    # 共享 guide
```

**`index.md` 是关键**：每层 spec 目录的 `index.md` 是单行目录 + 简短理由。brainstorm/research phase 只读 `index.md`（路径 + 理由），把相关条目写进 `implement.jsonl` / `check.jsonl`。具体 spec 内容文件只在 jsonl 列出来时才会被加载。这是按需注入的关键 —— 50 万行仓库 + 50 个 spec、当前任务只加载 4 个，对 AI 来说就跟 4 文件项目差不多。

**受保护路径**：`trellis update` 永远不会覆盖 `.trellis/spec/frontend/`、`.trellis/spec/backend/`、`.trellis/spec/<package>/`。模板更新只会改 spec 骨架（如新版的 `guides/`）。

## Hook 类型清单

Hook-capable 平台上 Trellis 注册的 hook：

| Hook 事件                                      | 触发时机                      | Trellis 行为                                                                |
| ---------------------------------------------- | ----------------------------- | --------------------------------------------------------------------------- |
| `SessionStart`                                 | 新会话开启（startup matcher） | 注入 workflow + 开发者 + git + active tasks + spec 索引（~16.7 KB payload） |
| `UserPromptSubmit`                             | 每条用户消息                  | 注入 `<workflow-state>` 面包屑（按当前 task status 给 ~200 字节提示）       |
| `PreToolUse:Agent` / Bun plugin `chat.message` | sub-agent 启动前              | 按 jsonl 注入 spec / prd 给 sub-agent（hook-push 平台）                     |

**关键 Python 脚本**（实际 hook 实现）：

| 脚本                         | 入口                          |
| ---------------------------- | ----------------------------- |
| `session-start.py`           | SessionStart                  |
| `inject-workflow-state.py`   | UserPromptSubmit              |
| `inject-subagent-context.py` | PreToolUse:Agent / Bun plugin |

> 0.4 时期的 `ralph-loop.py`（SubagentStop hook，做 check agent 的重试循环）已在 0.5 移除 —— check sub-agent 自带内部重试循环替代。

各平台的 hook 落点（`.claude/hooks/` / `.cursor/hooks/` / `.opencode/plugin/` / `.codex/hooks/` 等）共用同一组 Python 实现（`shared-hooks/`），各平台只是输出 adapter 不同。Fork 工作流改 `workflow.md` 一个文件即可，几乎不需要碰 Python。

## 关键脚本与路径速查

### 脚本

| 脚本                | 用途                                                                |
| ------------------- | ------------------------------------------------------------------- |
| `task.py`           | 任务管理（16 subcommand，见上）                                     |

| 路径                                    | 用途                                                          |

| `.trellis/workflow.md`                  | 工作流单一事实源                                              |
