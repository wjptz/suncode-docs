## v0.6

### Spec → Wiki 概念迁移

参考 [Karpathy 的 LLM Wiki 模式](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)，把 `spec` 概念扩展成更通用的"项目记忆 / 知识库"层。当前 `.trellis/spec/` 只承载 coding 约束，但用户实际想往里塞的东西远超这个范围 —— 决策日志、用户访谈结论、领域知识、跨任务的"为什么这样设计"…… 这些塞 spec 不合适，单写 PRD 又留不住。

同时借用 Karpathy 模式的**互相链接 + 持续维护**特性：wiki 不是一堆孤立 markdown，而是带 wikilinks、带交叉引用、带过期标记的活文档。spec 收窄到纯 coding 约束本身，wiki 接管更广义的项目记忆。

### Trace + Session insight

**Trace** 是跨任务可回放历史 —— 哪个 task 改了哪些文件、为什么改、引用了哪些 spec、最终被谁 review。当前的 journal 是个人会话脉络，trace 是跨开发者、跨任务的项目级活动流，方便事后回溯一个改动是怎么发生的。

**Session insight** 是配套的自动提炼层：每个 task 在 `finish-work` 时自动从 PRD / diff / journal 里提取关键决策、踩过的坑、不变量等沉淀点，写回到 wiki（见上面的 Spec → Wiki 概念迁移）对应位置 —— 不依赖开发者主动记得"我得把这个经验记下来"。Trace 是流水账，session insight 是流水账上的高亮和摘要。

### Auto runner

多个 Task 讨论完之后自动跑完 `implement` → `check` → `finish-work` → 下一个 task 的全流程，不用每个 phase 手动 `/trellis:continue`。给信任 Trellis 的团队一条更短的路径。

### Hook 更加可配置化

hook 行为通过配置文件控制开关，可能考虑跟当前 session 绑定（不同会话可以开启不同的 hook 组合）。同时把 hook 注入内容从 Python 脚本里的硬编码移到 `workflow.md`，之后改 hook 文案就是改 markdown，不再需要碰源码。

## v0.7

### 更好的记忆系统

在 0.6 wiki 层基础上做团队级 memory（不只是个人 journal）：session insight 自动写回 wiki，跨 task / 跨成员可检索，旧仓库 / 大代码库的"找上下文"成本显著下降。这是用户访谈里反复出现的最强信号。

### TDD 模板

Optional test-first 工作流模板，把 `implement → check` 改为 `test → implement → check`。当前 `trellis-check` 是事后检查；TDD 切换通过 `trellis-meta` skill 提供，不替换默认轻量工作流。

### Brainstorm 加强

让 brainstorm 在问业务问题之前先做更深入的代码 / 历史调研，把问题留给真正不能从代码推断的部分。当前 brainstorm 容易"问得太浅"。

### 中文本地化

CLI 输出、内置 skill / sub-agent prompt、默认模板的中文版本。`.trellis/spec/` 和 `workflow.md` 本身已经是模型无关 markdown，写中文工作正常 —— 这次补的是 Trellis 自带的英文文案。
