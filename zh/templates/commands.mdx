## 开发命令

### 代码审查

`.claude/commands/dev/review.md`：

```markdown
审查我刚写的代码。

检查：

1. **类型安全** - 类型对吗？有该具体化的 any 吗？
2. **错误处理** - 错误被正确捕获和处理了吗？
3. **边界情况** - 什么输入可能破坏这段代码？
4. **性能** - 有明显的低效吗？
5. **安全** - 输入验证、SQL 注入、XSS？

每个问题：

- 解释哪里错了
- 展示怎么修
- 评级严重性：critical / warning / suggestion

如果代码没问题，就说没问题。不要发明问题。
```

### 快速测试

`.claude/commands/dev/test.md`：

```markdown
给我刚写的代码写测试。

用项目的测试框架。测试文件放源文件旁边，用 .test.ts 扩展名。

覆盖：

- 正常路径（正常使用）
- 边界情况（空输入、null、边界值）
- 错误情况（无效输入、失败）

测试要专注。每个测试一个行为。用描述性测试名。

写完后跑测试。修复失败的。
```

### 调试助手

`.claude/commands/dev/debug.md`：

```markdown
帮我调试这个问题：$ARGUMENTS

步骤：

1. 理解期望 vs 实际行为
2. 找出 bug 可能在哪
3. 加日志或断点缩小范围
4. 找到根本原因
5. 提出修复方案

不要猜。跟踪代码路径。展示你的推理。
```

## Git 命令

### 智能提交

`.claude/commands/git/commit.md`：

```markdown
为我暂存的改动准备提交。

1. 跑 git diff --cached 看暂存的改动
2. 写 conventional commits 格式的 commit message：
   - feat: 新功能
   - fix: bug 修复
   - docs: 文档
   - refactor: 不修 bug 也不加功能的代码改动
   - test: 加或修测试
   - chore: 维护

3. 主题保持 72 字符以内
4. 如果改动需要解释就加正文

提交前给我看 message。我批准才提交。
```

### PR 描述

`.claude/commands/git/pr.md`：

```markdown
给当前分支写 PR 描述。

1. 跑 git log main..HEAD --oneline 看提交
2. 跑 git diff main...HEAD --stat 看改动的文件

结构：

## 概要

这个 PR 做什么？一段话。

## 改动

- 具体改动的列表

## 测试

怎么测的？审查者应该检查什么？

## 截图

（如果有 UI 改动）

保持简洁。审查者很忙。
```

## 文档命令

### API 文档

`.claude/commands/docs/api.md`：

```markdown
给这个 API 端点写文档：$ARGUMENTS

包含：

- HTTP 方法和路径
- 请求体（带类型）
- 响应格式（成功和错误）
- 认证要求
- 请求和响应示例

格式用 markdown。放到合适的文档位置。
```

## 部署命令

### 部署前检查

`.claude/commands/deploy/check.md`：

```markdown
跑部署前检查：

1. **测试**：npm test
2. **类型**：npm run typecheck
3. **Lint**：npm run lint
4. **构建**：npm run build

跑每个命令。报告结果。

如果有失败：

- 显示错误
- 建议修复
- 修好前不要继续

全通过就确认可以部署。
```

## 使用这些命令

1. 创建文件夹：`mkdir -p .claude/commands/dev`
2. 复制命令文件
3. 用 `/dev:review`、`/git:commit` 等使用

按你的项目定制：

- 改测试框架命令
- 调整 commit message 格式
- 加项目特定的检查
