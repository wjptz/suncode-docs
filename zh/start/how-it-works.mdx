跟着一个真实请求从头走一遍。看完你会知道一条 prompt 进会话之后，背后哪些 hook、skill、sub-agent、文件被串起来，怎么走到最后的 commit。

> 这页是使用视角。组件实现细节、扩展方式、fork 改造，看 [架构全景](/zh/advanced/architecture)。

## 1. 你打开会话

打开 Claude Code、Cursor、Codex 或任何支持的 AI coding 平台，进入有 `.trellis/` 的项目目录。

支持 hook 的平台会自动触发 SessionStart hook，把以下内容塞进 AI 的初始 context：

- 当前 git status、最近 commit、当前分支
- `.trellis/spec/` 目录树（团队规范库）
- 当前 session 的 active task pointer（如果有）
- `.trellis/workflow.md` 的 Phase Index 和完整 step 走读

不同平台用不同机制：

| 平台                                                | 注入方式                                   |
| --------------------------------------------------- | ------------------------------------------ |
| Claude Code, Cursor, OpenCode, CodeBuddy, Droid, Pi | SessionStart hook 在 AI 启动前主动 push    |
| Codex, Copilot, Gemini, Qoder                       | sub-agent 通过 prelude 自己拉（hook 可选） |
| Kilo, Antigravity, Windsurf                         | 没有 hook 原语，靠主动加载 skill           |

终端打开那一刻 AI 已经知道这是 Trellis 项目、当前 active task 是哪个、`spec/` 里有哪些规范文件。你不用 `cat workflow.md`。

## 2. 你发了第一句 prompt

每条用户消息都触发 `UserPromptSubmit` hook。这个 hook 做一件事：读 `task.json.status`，找到 `.trellis/workflow.md` 里对应的 `[workflow-state:STATUS]` block，包进 `<workflow-state>...</workflow-state>` 注入到当前 turn。

AI 每条消息都会看到这个 breadcrumb。它告诉 AI：现在在哪个 phase（no_task / planning / in_progress / completed）、下一步该干什么、哪些步骤是 `[required]` 不能跳的、用户哪些短语能让 AI 跳过流程。

```text
<workflow-state>
Status: no_task
A 直接答 — 纯问答 / 解释 / 查询 / 闲聊...
B 建任务 — 任何实施 / 改动 / 构建 / 重构工作...
C inline 改 — 仅本轮，B 的 escape hatch...
</workflow-state>
```

AI 不靠"记得读" workflow.md。每轮 hook 都重新喂一份当前 phase 的提醒。

## 3. AI 决定走哪条路

读完 prompt 和 breadcrumb，AI 在 A、B、C 三条路里挑一条：

- **A 直接答**：纯问答、解释、文档查询、闲聊。判定标准是不写文件 + 一句话能答清 + 仓库读不超过 2 个文件。回完结束。
- **B 建任务**：你说的是要做事——实施、改动、构建、重构。看上去多小都进 Trellis 流程。
- **C inline 改**：你显式说了 "直接改" / "skip trellis" / "小修一下" 之类的关键词。本轮跳过流程直接改，下轮回到 B。

下面的 step 都是 B 路径展开。

## 4. 建任务

AI 跑：

```bash
python3 ./.trellis/scripts/task.py create "<title>" --slug <name>
```

发生三件事：建 `.trellis/tasks/<MM-DD-name>/` 目录，写 `task.json`（status=planning），把当前 session 的 active task pointer 自动指向新任务。第三件事意味着不需要单独跑 `start`。

下一轮 breadcrumb 切到 `[workflow-state:planning]`，告诉 AI：用 `trellis-brainstorm` skill 跟用户讨论需求，写 `prd.md`；写完整理 `implement.jsonl` 和 `check.jsonl`；最后 `task.py start`。

接下来进 brainstorm。AI 加载 `trellis-brainstorm` skill，跟你 Q&A。一次问一个问题，每次回答完立刻把内容写进 `prd.md`。需要调研第三方库或跨平台对比，AI 会派 `trellis-research` sub-agent，sub-agent 自己写到 `{task}/research/<topic>.md`，主对话不污染。

prd 写完进 Phase 1.3，整理两个 jsonl 文件：

- `implement.jsonl`：implement sub-agent 写代码时要注入的 spec 和 research 文件
- `check.jsonl`：check sub-agent 验收时要看的 spec

只列 spec 路径和 research 路径。代码文件不进 jsonl，sub-agent 自己读。

最后激活任务：

```bash
python3 ./.trellis/scripts/task.py start <task-dir>
```

`status` 从 planning 翻到 in_progress，breadcrumb 切到 `[workflow-state:in_progress]`，开始实施。

## 5. Sub-agent 实施

AI 派 `trellis-implement` sub-agent。重点：不是让主对话写代码，是把活分给 sub-agent。

Hook 自动给 sub-agent 注入 `prd.md` 全文、`implement.jsonl` 列的所有 spec 文件、`info.md`（如果有）。注入路径有三种：

| 类型         | 平台                                                | 机制                                                              |
| ------------ | --------------------------------------------------- | ----------------------------------------------------------------- |
| Push hook    | Claude Code, Cursor, OpenCode, CodeBuddy, Droid, Pi | `inject-subagent-context.py` hook 在 sub-agent 启动前 push        |
| Pull prelude | Codex, Copilot, Gemini, Qoder                       | sub-agent 定义里写好 prelude，sub-agent 自己 `Read`               |
| Skill 内联   | Kilo, Antigravity, Windsurf                         | 没有 sub-agent 原语，主对话加载 `trellis-before-dev` skill 自己读 |

Sub-agent 写代码、跑 lint 和 typecheck，结束。它不 commit。commit 的边界在人这边。

## 6. 自动验收

AI 派 `trellis-check` sub-agent。同样的注入机制，这次走 `check.jsonl`。

Check sub-agent 干四件事：对照 spec 复核所有改动、跑 lint / typecheck / 测试、看跨层一致性（改动跨 layer 时）、发现问题直接修而不是只报告。

跑通进 step 7。跑不通报 finding，回 step 5 修，再 check。

## 7. 沉淀和提交

加载 `trellis-update-spec` skill。AI 自问：本次任务有没有发现值得写回 `spec/` 的东西。例如踩了某个坑、发现某个新 pattern、定了某个技术决策——写进 `spec/<package>/<layer>/*.md`，下次别的任务的 sub-agent 就能自动读到。

主 agent 主导 commit（这是 Phase 3.4，required）。先把 commit 计划告诉你，等你确认后跑 `git commit`。它不 push。push 是你的事。

最后 AI 提示你跑 `/trellis:finish-work`。这一步会把任务 archive 到 `tasks/archive/YYYY-MM/`，再把 session 写进 `.trellis/workspace/<dev>/journal-N.md`。

## 沉淀都在文件里

整套流程跑完，所有产出都落到了文件：

- 你的 PRD、research、info → `tasks/<task>/`
- 抽提的规范 → `spec/`
- 个人决策日志 → `workspace/<dev>/journal-N.md`
- 归档的任务 → `tasks/archive/YYYY-MM/<task>/`

下次会话开始 SessionStart hook 把这些都索引到 AI 的初始 context。整个团队的 sub-agent 都能读到，不光是写它的那个开发者。

## 为什么这样设计

Hook 在 AI 看到 prompt 之前就把规范 push 进去——AI 不用记得自己读。Sub-agent 三层分工，研究、写、验收互相独立，每个 agent 范围都有限。JSONL 配置每个任务真正用到的 spec，主对话 context 不爆炸。AI 不 commit，commit 是 PR review 的入口。你写过的规范沉淀回 `spec/`，下次别人写代码时 sub-agent 自动读得到。
