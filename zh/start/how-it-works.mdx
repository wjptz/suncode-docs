title: 'How It Works'
---

跟着一个真实请求从头走一遍，看 Trellis 在每一步到底做了什么。看完你会知道：每条 prompt 进入会话之后，背后哪些 hook、skill、sub-agent、文件被串起来，最终产出代码并落到 commit。

> 这页讲使用视角下的工作流。想了解每个组件的实现细节、怎么扩展、怎么 fork，看 [架构全景](/zh/advanced/architecture)。

## Step 1：你打开会话

打开 Claude Code / Cursor / Codex / 任何支持的 AI coding 平台，进入有 `.trellis/` 的项目目录。

**SessionStart hook 自动触发**（hook-capable 平台），把这些塞进 AI 的初始 context：

- 当前 git status、最近 commit、当前分支
- `.trellis/spec/` 索引（团队规范库的目录树）
- 当前 session 的 active task pointer（如果有）
- `.trellis/workflow.md` 的 Phase Index + 完整 step 走读

不同平台用不同机制：

| 平台                                                     | 注入方式                                     |
| -------------------------------------------------------- | -------------------------------------------- |
| Claude Code / Cursor / OpenCode / CodeBuddy / Droid / Pi | SessionStart hook，AI 启动前主动 push        |
| Codex / Copilot / Gemini / Qoder                         | sub-agent 用 pull-based prelude（hook 可选） |
| Kilo / Antigravity / Windsurf                            | 没有 hook 原语，靠主动加载 skill             |

**结果**：你打开终端时，AI 已经知道这是个 Trellis 项目、当前没有 active task、规范都放在 `spec/` 哪些路径。你不用手动 `cat workflow.md`。

## Step 2：你发了第一句 prompt

每条用户消息都触发 `UserPromptSubmit` hook（hook-capable 平台）。这个 hook 做一件事：

- 读 `task.json.status`（如果有 active task）
- 找到 `.trellis/workflow.md` 里对应的 `[workflow-state:STATUS]` block
- 把 block 内容包进 `<workflow-state>...</workflow-state>` 注入到 AI 的当前 turn

这是 AI 每轮都看到的"工作流路标"。它告诉 AI：

- 现在在哪个 phase（无任务 / 规划中 / 实施中 / 已完成）
- 下一步该干什么
- 哪些步骤是 `[required]` 不能跳的
- 用户什么短语能让你跳过流程

```text
<workflow-state>
Status: no_task
A 直接答 — 纯问答 / 解释 / 查询 / 闲聊...
B 建任务 — 任何实施 / 改动 / 构建 / 重构工作...
C inline 改 — 仅本轮，B 的 escape hatch...
</workflow-state>
```

**结果**：AI 不靠"记得"读 workflow.md，每轮都被 hook 主动喂一次当前 phase 的提醒。

## Step 3：AI 决定走哪条路

AI 看完你的 prompt + 当前 breadcrumb，决定走 **A / B / C** 三条路之一：

- **A 直接答** — 纯问答、解释、文档查询、闲聊。判定是"不写文件 + 一句话能答清 + 仓库读不超过 2 个文件"。直接回答完事。
- **B 建任务** — 你描述的是实施 / 改动 / 构建 / 重构工作。无论看上去多小，都进 Trellis 流程。
- **C inline 改** — 你显式说了"直接改" / "skip trellis" / "小修一下" 等关键词，本轮跳过流程直接改。仅本轮生效，下轮还是默认走 B。

下面的 step 都是 B 路径展开。

## Step 4：建任务

AI 跑：

```bash
python3 ./.trellis/scripts/task.py create "<title>" --slug <name>
```

发生三件事：

1. `.trellis/tasks/<MM-DD-name>/` 目录创建
2. `task.json` 写入 `status: "planning"`
3. **当前 session 的 active task pointer 自动指向新任务**（不用单独跑 `start`）

下一轮 breadcrumb 自动切到 `[workflow-state:planning]`，告诉 AI：

> 接下来用 `trellis-brainstorm` skill 跟用户讨论需求，写 prd.md；写完整理 `implement.jsonl` / `check.jsonl`；然后 `task.py start`。

**brainstorm 阶段**：AI 加载 `trellis-brainstorm` skill，跟你 Q&A。一次问一个问题，每次回答完立刻把内容写进 `prd.md`。需要调研第三方库 / 跨平台对比，AI 会派 `trellis-research` sub-agent —— sub-agent 自己写到 `{task}/research/<topic>.md`，主对话不污染。

**Phase 1.3 jsonl curation**：prd 写完 AI 整理两个文件：

- `implement.jsonl` — 列出 implement sub-agent 写代码时要注入的 spec / research 文件
- `check.jsonl` — 列出 check sub-agent 验收时要看的 spec

只列 spec 文件路径和 research 文件路径，不列代码文件——代码 sub-agent 自己读。

**激活任务**：

```bash
python3 ./.trellis/scripts/task.py start <task-dir>
```

`status: "planning" → "in_progress"`，breadcrumb 切到 `[workflow-state:in_progress]`，下一步开始实施。

## Step 5：Sub-agent 实施

AI 派出 `trellis-implement` sub-agent。**关键**：你不是让主对话写代码，是把活分给 sub-agent。

**Hook 自动注入** sub-agent 的初始 context：

- `prd.md` 全文
- `implement.jsonl` 里列的所有 spec 文件
- `info.md`（如果有）

注入路径有三种，看平台：

| 类型         | 平台                                                     | 机制                                                              |
| ------------ | -------------------------------------------------------- | ----------------------------------------------------------------- |
| Push hook    | Claude Code / Cursor / OpenCode / CodeBuddy / Droid / Pi | `inject-subagent-context.py` hook 在 sub-agent 启动前 push        |
| Pull prelude | Codex / Copilot / Gemini / Qoder                         | sub-agent 定义里写好 prelude，sub-agent 自己 `Read`               |
| Skill 内联   | Kilo / Antigravity / Windsurf                            | 没有 sub-agent 原语，主对话加载 `trellis-before-dev` skill 自己读 |

Sub-agent 写代码、跑 lint / typecheck，结束。**写完不 commit** —— commit 是人的边界。

## Step 6：自动验收

AI 派 `trellis-check` sub-agent。同样有 hook 注入 `check.jsonl` 列的 spec。

check sub-agent 干四件事：

1. 对照 spec 复核所有改动
2. 跑 lint / type-check / 测试
3. 跨层一致性检查（改动跨多个 layer 时）
4. 发现问题**直接修**，不光报告

跑通 → 进 Step 7。跑不通 → 报 finding，回 Step 5 修，再 check。

## Step 7：沉淀 + 提交

**3.3 update-spec**：AI 加载 `trellis-update-spec` skill，问"本次任务有没有新发现值得写回 `spec/`"。例：踩了某个坑、发现某个新 pattern、定了某个技术决策——写进 `spec/<package>/<layer>/*.md`，下次别的任务的 sub-agent 自动读得到。

**3.4 commit（required）**：主 agent 主导 commit。先把 commit 计划告诉你，等你确认后跑 `git commit`。**不会 push**——push 是你的事。

**3.5 收尾**：AI 提示你跑 `/trellis:finish-work`：

- `task.py archive` 把任务搬到 `tasks/archive/YYYY-MM/`
- 写 session journal 到 `.trellis/workspace/<dev>/journal-N.md`

任务结束。

## 落到文件，不落到对话

整套流程跑完，**沉淀都在文件里**：

- 你的 PRD / research / info → `tasks/<task>/`
- 抽提的规范 → `spec/`
- 个人决策日志 → `workspace/<dev>/journal-N.md`
- 归档的任务 → `tasks/archive/YYYY-MM/<task>/`

下个会话开始时，SessionStart hook 把这些都索引到 AI 的初始 context。**全队的 sub-agent 下次都能读到**——不光是你自己。

## 为什么这样设计

每一步都不是随便加的：

- **Hook 在 AI 看 prompt 之前主动 push** → AI 不靠"记得读 spec"
- **Sub-agent 三层（research / implement / check）分工** → 每个 agent 范围有限，不会一个 agent 干所有事，互相约束
- **JSONL 配置而不是 dump 全部 spec** → context 不爆炸，每个任务只注入它真正用到的
- **AI 不 commit** → commit 是 PR review 的入口，必须人决定
- **沉淀回 `spec/`** → 一次解决永久受益，全团队复利
